# 矢羽プラグイン

## 機能

現在のレコードの状況を、矢羽根(チェブロン状の矢印を並べた進捗インジケーター)でレコードのヘッダーメニューに
表示する。ドロップダウン・ラジオボタン・プロセス管理のステータスのいずれかの値をもとに、対応する矢羽根が
アクティブになる。

- 矢羽根ウィジェット(以下「ウィジェット」)を複数持てる。
- ウィジェットごとに、対象フィールド(ドロップダウン/ラジオボタン、またはプロセス管理のステータス)・
  矢羽根の並び順(ステップの配列。フィールドの選択肢すべてを含める必要はない)・デザイン(配色プリセット)を
  設定できる。

## 状態モデル(確定・3状態)

矢羽根UIとして一般的な「完了・現在・未着手」の3状態モデルを採用した(元メモは「値によって矢羽根がactiveに
なる」としか書いていないが、進捗表示という用途上、現在値より前のステップも視覚的に区別できたほうが実用的と
判断、`判断記録.md`参照)。

- `DONE`(完了): 現在値のステップより前にあるステップ
- `ACTIVE`(現在): 現在値と一致するステップ
- `PENDING`(未着手): 現在値のステップより後、または現在値がステップの並びに含まれない場合のすべてのステップ

`js/lib/arrow-state.js`の`computeArrowStates(steps, currentValue)`がこのロジックを実装する。

## 対象フィールド(確定)

- **ドロップダウン(DROP_DOWN)・ラジオボタン(RADIO_BUTTON)**: 通常のフィールドとして`kintone.app.getFormFields()`
  で列挙する。
- **プロセス管理のステータス**: kintoneのフィールド形式の仕様上、ステータスフィールドは`record['ステータス']`
  という固定のフィールド名(フィールドコードではない)でアクセスする([フィールド形式](https://cybozudev.zendesk.com/)
  のドキュメント記載の通り、カテゴリー・ステータス・作業者の3フィールドのみフィールド名でアクセスする)。
  本プラグインでは`sourceType: 'STATUS'`を選ぶと、この固定名でレコードから値を読む。設定画面でステータス名の
  選択肢一覧を取得する際は、JavaScript API(`kintone.app.getStatus()`)がプラグイン設定画面では
  利用できない(利用できる画面がレコード一覧・追加・編集・詳細・グラフ画面のみ)ため、REST API
  (`GET /k/v1/app/status.json`)を`kintone.api()`経由で呼ぶ(`js/config.js`参照)。

## 矢羽根の並び順(ステップ)は手動設定(確定)

対象フィールドの選択肢すべてを自動的に矢羽根にするのではなく、アプリ管理者が設定画面で使いたい選択肢を
選んで並び順を指定する(選択肢の一部だけを矢羽根にする、選択肢の定義順と異なる順序で並べる、といった
柔軟性のため)。ステップの値は自由入力ではなく、対象フィールド(`kintone.app.getFormFields()`の
`options`)またはプロセス管理のステータス(`kintone.app.getStatus()`の`states`)から取得した選択肢を
`<select>`で選ぶ方式にしている。値の一致判定(`arrow-state.js`)に使う文字列を手入力させると、
対象フィールドの選択肢の表記と食い違うタイプミスが起きやすいため。対象フィールド・対象種別の変更後、
既存のステップ値が新しい選択肢に含まれなくなった場合は選択肢から黙って消さず、「(選択肢に存在しません)」
と表示したまま残して管理者に気づかせる(`js/config.js`の`renderStepList()`)。

## 描画位置(確定)

「基本レコードのヘッダーメニューにする」という元メモの通り、`kintone.app.record.getHeaderMenuSpaceElement()`
(レコードのパンくずリストとメニューの間の領域)に矢羽根UIを描画する。このAPIはPC専用(モバイルには対応
APIがない)。

## 発動する画面・モバイル対応(確定・スコープ限定)

`kintone.app.record.getHeaderMenuSpaceElement()`が利用できる画面(レコード詳細・追加・編集画面、PC専用)で
発動する。モバイルには同等のAPI(レコード単位のヘッダーメニュー領域を取得するAPI)が存在しないため、
本プラグインはモバイル非対応とする(`判断記録.md`参照)。

## デザイン(配色プリセット、確定)

`DEFAULT`・`BLUE`・`GREEN`・`ORANGE`の4種類の配色プリセットから選べる。プリセットの選択はCSSクラス名の
切り替えのみで実現し(`js/lib/design-preset.js`の`resolveDesignClass()`)、未知の値は`DEFAULT`にフォールバック
する。

## 設定画面

`kintone.plugin.app.setConfig()`にのみ保存する。

- ウィジェットを追加・削除できる。ウィジェットごとに:
  - 対象種別(ドロップダウン/ラジオボタン から選択するフィールド、またはプロセス管理のステータス)
  - 矢羽根のステップ(選択肢を追加・削除・並び替え)
  - デザインプリセット
- 保存時に`js/lib/config-validation.js`でチェックする(対象未選択、ステップ0件等)。

## TDD

`src/js/lib/`配下の純粋ロジックはJestでユニットテストする(`pnpm test`)。

- `arrow-state.js` — ステップの配列+現在値から、各ステップの状態(`DONE`/`ACTIVE`/`PENDING`)を求める
- `design-preset.js` — デザインプリセット名からCSSクラス名を解決する(未知の値はデフォルトにフォールバック)
- `config-store.js` — `kintone.plugin.app.getConfig()`/`setConfig()`のペイロードの読み書きとデフォルト値
- `config-validation.js` — 設定(ウィジェットの配列)のバリデーション

kintone依存のグルーコード(`desktop.js`/`config.js`、矢羽根UIのDOM描画・
`kintone.app.record.getHeaderMenuSpaceElement()`呼び出しを含む。モバイル非対応のため`mobile.js`は作成しない)は
今回のタスクスコープでは実環境テスト(Puppeteer)を行わない(別タスク、e2eは後回し)。

## 実装

kintoneドキュメントMCPを参照しながら実装した。`kintone.app.record.getHeaderMenuSpaceElement()`がPC専用で
レコード詳細・追加・編集画面のみ利用できること、プロセス管理のステータスフィールドがフィールドコードではなく
固定のフィールド名`ステータス`でアクセスすることを確認済み。

設定画面のステップ選択肢取得では、当初JavaScript API `kintone.app.getStatus()`を使う設計だったが、
MCPドキュメントに「利用できる画面」がレコード一覧・追加・編集・詳細・グラフ画面のみでプラグイン設定画面が
含まれないと明記されており、実機の設定画面で呼び出すと実際に`kintone.app.getStatus is not a function`
になることをPuppeteerでのスクリーンショット撮影時に確認した(実装前のMCP参照だけでは気づけず、実機確認で
発覚)。そのため設定画面ではREST API `GET /k/v1/app/status.json`を`kintone.api()`経由で呼ぶ方式に修正した
(CLAUDE.md開発方針3のとおり、JavaScript APIで実現できない場合のみRESTを使い、kintone自身への呼び出しは
`kintone.api()`を使う)。このREST APIレスポンスはドキュメントどおり`{ enable, states, actions, revision }`
というプロパティ名でラップされた値で返る(`states`はプロセス管理未設定のアプリでは`null`)。
セキュアコーディングガイドラインでのリスクチェックは`security-checklist.md`を参照。
