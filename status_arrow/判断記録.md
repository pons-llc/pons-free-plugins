# 判断記録

`plugin_idea.md`の元メモ(「矢羽プラグイン」の項)は数行と短く、実装時に暫定判断が必要だった事項を
記録する。作業は止めずに進めた。

## 1. 状態モデルを「完了/現在/未着手」の3状態にした

- **問題**: 元メモは「値によって矢羽根がactiveになる」としか書いておらず、現在値以外のステップを
  区別するかどうかが不明だった(単純な2状態: アクティブ/非アクティブ、と読むこともできる)。
- **選んだ暫定対応**: 「矢羽根」という進捗インジケーターの一般的なUI表現(cybozu公式サンプル等でも
  見られる「完了(色付き)・現在(強調)・未着手(グレー)」の3状態表示)に合わせ、3状態モデルを採用した。
  2状態だと単なるタブ切り替えのような見た目になり、「現在の状況が分かりやすくする」という元メモの
  目的(進捗の可視化)を十分に満たせないと判断した。
- **要ユーザー確認**: 2状態(現在値のみactive、それ以外はすべて同じ非アクティブ表示)を意図していた
  場合は、`js/lib/arrow-state.js`の`computeArrowStates()`を単純化する変更が必要。

## 2. モバイル非対応とした

- **問題**: 元メモに画面の指定はなかったが、「基本レコードのヘッダーメニューにする」という描画位置の
  指定に対応するJavaScript API(`kintone.app.record.getHeaderMenuSpaceElement()`)がPC専用で、
  モバイルには同等のAPI(レコード単位のパンくずリスト直下の領域を取得するAPI)が存在しないことを
  MCP経由で確認した。
- **選んだ暫定対応**: モバイル非対応とした(`mobile.js`を作成しない)。モバイルの「ヘッダーの下側の
  要素を取得するAPI」はレコード単位ではなく画面全体で共通のため、レコードごとに異なる矢羽根を
  出し分ける用途には適さないと判断した。
- **要ユーザー確認**: モバイル対応が必要な場合、スペースフィールドを別途アンカーとして使う方式
  (`tab_layout`と同様の`getSpaceElement()`方式)への設計変更が必要。

## 3. 矢羽根のステップは選択肢の自動反映ではなく手動設定にした

- **問題**: 元メモに、対象フィールドの選択肢をそのまま矢羽根にするのか、選択肢の一部・別順序を
  設定できるようにするのかの記載がなかった。
- **選んだ暫定対応**: 手動設定(選択肢を追加・削除・並び替え)にした。ドロップダウン/ラジオボタンの
  選択肢がそのままステップの理想的な並び順・完全な集合と一致するとは限らない
  (例: 「保留」のような選択肢を矢羽根から除外したい、選択肢定義の追加順と業務上の進行順が異なる、等)
  ため、柔軟性を優先した。
- **要ユーザー確認**: 選択肢を自動的にすべて反映してほしい場合は、設定画面に「フィールドの選択肢から
  一括インポート」ボタンを追加する拡張が考えられる(ロジック側の変更は不要)。

## 4. ウィジェットを複数持てるようにした

- 元メモは単一の矢羽根UIのイメージだが、`subtable_lookup`等と同様、複数のウィジェット(例:
  「対応状況」と「承認状況」を別々の矢羽根で表示したい)を持てるようにした。
