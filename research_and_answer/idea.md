# 自治体照会回答作成システム

## 1. アプリ構成

本システムは、わずか2つのアプリで仮想的に多様な照会フォームを実現するパッケージ構成となっています。

### 1.1 依頼アプリ
照会内容（フォーム項目・レイアウトなど）を作成し、全体を管理するアプリです。
- **アプリ設計**: [設計情報](./backup/app_202)
- **カスタマイズファイル**: [./customise/request/](./customise/request/)
  - `make.js`: 動的フォーム作成・プレビュー機能、JSON形式へのレイアウト保存
  - `analisys.js`: 集計・分析連携

### 1.2 回答アプリ
依頼アプリの設計情報（JSON）を参照し、各部署の回答者向けフォームや一覧、ダッシュボードを動的に生成するアプリです。
- **アプリ設計**: [設計情報](./backup/app_203)
- **カスタマイズファイル**: [./customise/answer/](./customise/answer/)
  - `virtual.js`: JSONメタデータに基づく仮想一覧（「集計リスト」）の動的描画
  - `answer.js`: 仮想フォームの生成と回答制御
  - `analysis.js`: 回答データの集計・ダッシュボード機能

## 2. 実現したいこと

### 2.1 現状の課題
自治体では、各部署から各部署へ「照会」というかたちで調査が頻繁に行われ、各部署はそれに回答しています。  
しかし、現在どのような照会が依頼されているのか、各部署の回答状況がどうなっているのかが「見える化」されていないため、期限が守られないといった問題が起きています。  
さらに、kintoneを活用して課題解決を試みても、結果的に「部署ごと」「照会ごと」に使い捨てのアプリを作成してしまい、タスクがサイロ化するだけでなく、アプリの「野良化」やkintoneのアプリ数上限に達してしまうという新たな課題が生じています。

### 2.2 解決策
これらを踏まえ、kintone内でノーコードによって柔軟に照会フォームを作成でき、**「依頼アプリ」「回答アプリ」という2つのアプリだけで、あらゆる照会業務に汎用的に対応可能**なアプリケーションパッケージ（本システム）を作成しました。

## 3. 主要な機能（仕組み）

1. **データスキーマの仮想化と動的フォーム生成**
   - 回答アプリには予め、各種データタイプ（文字列、数値、日時、時間など）に対応した汎用的・非表示の大量フィールドを用意しています。
   - 依頼アプリ側で「どのフィールドを使用してどのようなフォームにするか」を指定し、その構造をJSONとして保存します。
   - 回答アプリはそのJSONを読み込み、事前定義された非表示フィールドを動的にフォーム上に割り当てることで、1つのアプリで複数の一意なアプリを仮想的にシミュレートします。
2. **動的形式一覧とダッシュボード**
   - 仮想化された大量のフィールドを持つ回答アプリでも直感的にデータが閲覧できるよう、依頼データをもとに現在の照会に必要なカラムのみを動的生成し標準機能に依存しない「集計リスト」を描画します（virtual.js）。
   - 簡易的なダッシュボードによる集計・一覧出力も実装されています。

## 4. プラグイン化仕様(`src/`)

`research/`・`answer/`のカスタマイズファイル群を、1つのkintoneプラグインとして再実装する。
1つのプラグインを依頼アプリ・回答アプリの両方にインストールし、設定画面で「このアプリの役割」を
選択することで動作を切り替える。

### 4.1 設定画面(config)

| 設定キー | 内容 | 既定値 |
| --- | --- | --- |
| `role` | このアプリの役割(`request`=依頼アプリ / `answer`=回答アプリ)。必須 | (未選択) |
| `previewSpaceId` | (依頼)プレビュー描画先スペースの要素ID | `preview` |
| `formSpaceId` | (回答)仮想フォーム描画先スペースの要素ID | `form_space` |
| `listViewName` | (回答)集計リストを描画する一覧名 | `集計リスト` |
| `analysisViewName` | (回答)分析ダッシュボードを描画する一覧名 | `分析` |

| `answerAppId` | (依頼)回答アプリのアプリID(リレーション先) | (未入力) |
| `requestAppId` | (回答)依頼アプリのアプリID(リレーション元) | (未入力) |

フィールドコード(依頼アプリの`questions`テーブル・`json`・`condition_json`・`related_links`・
`related_rich_text`・参照テーブル`related`、回答アプリの`json`・`title`・`lookup`・予備フィールド
`text_*`/`multi_text_*`/`number_*`/`date_*`/`datetime_*`/`time_*`)は固定規約とし、設定画面では
変更しない。

### 4.1.1 必要項目の自動生成(設定画面のボタン)

アプリテンプレートの手動構築を不要にするため、設定画面の「必要な項目・一覧を自動作成」ボタンで、
動作テスト環境API(`/k/v1/preview/app/form/fields.json`・`form/layout.json`・`views.json`、いずれも
kintone_doc MCPで仕様確認済み)を使って不足分だけを冪等に追加する(既存フィールドは変更しない)。
反映は通常のプラグイン設定保存と同じく「アプリを更新」で行う。リレーションは設定画面で入力した
相手アプリIDを使って作成する。

- **依頼アプリ**: `title`・`description`・`requester`(組織選択=依頼元部署)・`recipients`
  (組織選択=依頼先部署)・`deadline`(日時=回答期限)・`attachment`(添付=関連添付ファイル)・
  `questions`テーブル(`order`/`question`/`question_detail`/`field_type`/`choice`/
  `question_width`/`mondatory`)・`condition_json`・`json`・`related_links`テーブル(`link_link`/
  `link_display_label`)・`related_rich_text`・プレビュー用スペーサー。`related`(関連レコード一覧、
  `このアプリのレコード番号 ↔ 回答アプリのlookup`)は回答アプリの**運用環境**に`lookup`が存在する
  場合のみ作成する(通常は回答アプリ側の再実行で自動追加されるため、こちらはフォールバック)。
- **回答アプリ**: `title`・`description`・`requester`(組織選択)・`deadline`(日時)・`json`
  (以上ルックアップの自動コピー)・`answer_department`(組織選択=回答部署、初期値
  `PRIMARY_ORGANIZATION()`=優先する組織)・`answer_status`(ラジオボタン=回答状況:
  未着手/対応中/回答済、初期値 未着手)・`attachment`(添付ファイル)・`lookup`(NUMBER+lookup設定。
  関連付け先=依頼アプリID、コピー元=依頼アプリのレコード番号、`title`/`description`/`requester`/
  `deadline`/`json`をコピー。組織選択→組織選択のコピーは実機E2Eで動作確認済み)・予備フィールド
  (`text_1..20`/`multi_text_1..10`/`number_1..10`/`date_1..5`/`datetime_1..5`/`time_1..5`)・
  回答フォーム用スペーサー・一覧2つ(集計リスト=カスタマイズ形式
  `<div id="virtual-table-div"></div>`+ページネーションあり、分析=表形式)。依頼アプリ側
  (運用環境)が未生成の場合はエラーで先に依頼アプリ側の実行+アプリ更新を促す。
- **実行順**: ①依頼アプリで実行→アプリ更新 → ②回答アプリで実行→アプリ更新 →
  ③回答アプリで再実行(依頼アプリへ`related`を自動追加)→依頼アプリを更新。冪等なので
  何度実行してもよい。**ルックアップ・関連レコード一覧の参照整合性チェックは相手アプリの
  運用環境(デプロイ済み)に対して行われる**ため、この順が必要(E2Eで「指定されたフィールド
  (code: lookup)が見つかりません」エラーとして実際に確認し、運用環境チェックに修正した)。
- 回答の集計リスト・分析では、ルックアップで照会からコピーされる「照会ごとに一定」の値
  (`title`/`description`/`requester`/`deadline`/`json`)は列・グラフに出さない。
  `answer_department`/`answer_status`は列・フィルター・グラフに出る。
- **制約**: カスタマイズ形式一覧の作成にはkintoneシステム管理権限が必要(REST API仕様)。権限が
  ない場合はエラーを表示して手動作成を案内する。集計リストの描画は`#virtual-table-div`が無い
  場合ヘッダースペースにフォールバックするため、表形式の「集計リスト」一覧でも動作する。
- **予備フィールドの非表示**: テンプレートでグループに隠す代わりに、レコード画面表示時に
  `kintone.app.record.setFieldShown()`(存在確認済み)で予備フィールドと`json`を非表示にする。

### 4.2 元カスタマイズからの変更点(分析まわりの仕様改善)

元の`analysis.js`は仕様に自信がないとのことだったため、プラグイン化にあたり以下を改善した。

1. **Chart.js(CDN)依存の廃止** — 外部スクリプト読み込みは開発方針9(外部通信禁止)とセキュア
   コーディングガイドラインに反するため、横棒/縦棒/ドーナツ/折れ線をvanilla JS + SVGで自前描画する
   (`js/lib/chart-lite.js`)。
2. **`confirm("全件分析を開始しますか？")`の廃止** — 分析ビューを開いたら自動で全件取得を開始し、
   ローディング表示と取得件数を出す。取得はレコード一覧の絞り込み条件(`getQueryCondition()`)を
   引き継いだカーソルAPI(500件/回)。
3. **グラフ表示の独立** — 旧仕様は「条件」チェック(絞り込みフィルターの表示)がグラフ表示を兼ねて
   いた。表示設定を「表」「条件」「グラフ」の3チェックに分離。
4. **「設定JSON」テキストエリアの廃止** — 表示設定(表示列・フィルター・グラフ種別・集計方法)は
   `localStorage`(アプリID+照会レコードID単位のキー)へ自動保存し、再訪時に復元する。保存するのは
   表示設定のみで認証情報等は含まない。
5. **グラフクリックでの絞り込みの修正** — 元コードは棒/円グラフのハンドラーが`onclick`(小文字)で
   Chart.jsに認識されず動いていなかった。自前描画で正しく実装。
6. **依頼アプリ側の分析(`research/analisys.js`)の廃止** — 全行コメントアウトされた死にコードのため
   移植しない。分析は回答アプリの「分析」ビューに一本化し、依頼アプリのレコード詳細の
   「集計・分析」ボタンから遷移する(従来どおり)。
7. **`kintone.app.getViews()`の戻り値の形をコメントで明記** — 戻り値は`{type, builtinType, name,
   id}`の**配列**(表示順)であり、REST `GET /k/v1/app/views.json`の`{views: {一覧名: ...}}`形とは
   異なる(kintone_doc MCPで確認済み。JavaScript API優先の方針どおりこのAPIを使用)。
8. **XSS対策** — 関連リンク(`related_links`)からリッチテキストを生成する際、URLは
   `http(s)`スキームのみ許可+属性エスケープ、表示ラベルはHTMLエスケープする(元コードは無検証で
   HTML連結していた)。
9. **数値KPIの合計/平均切替・CSV/JSONエクスポート(UTF-8 BOM対応)・PDF保存(`window.print()`)は維持。**

### 4.5 セットアップの前提(アクションボタン・アクセス権)

プラグイン自体が自動生成するのはフィールド・一覧・(可能な場合の)related関連レコード一覧までで、
以下の2点はプラグインの範囲外・kintoneの標準機能で管理者が別途設定する前提とする。設定画面や
公開サイトのセットアップ手順にはその旨を明記すること。

1. **回答の追加はプロセス管理のアクションボタン経由を想定** — 依頼アプリのプロセス管理で
   ステータス遷移時の「アクション」機能(別アプリへレコードを自動生成)を設定し、回答部署の
   担当者が依頼アプリのレコードからアクションボタンを押すことで回答アプリにレコード
   (`title`/`description`/`requester`/`deadline`等をコピー)を追加する運用を前提とする。
   プラグインはこのプロセス管理・アクション設定自体は自動生成しない(kintone標準のアプリ設定
   画面で行う)。
2. **回答アプリのアクセス権は「回答部署」「依頼元部署」に閲覧・編集権限を付与する設定を想定** —
   回答アプリのレコードのアクセス権を、`answer_department`(回答部署)または`requester`
   (依頼元部署)の組織に値が一致する場合のみ閲覧・編集を許可するよう設定する運用を前提とする
   (関係のない部署の回答が見えない/編集できないようにするため)。プラグインはアクセス権の設定
   自体は行わない(セキュリティ上、アクセス権はkintone管理者が画面またはREST APIで個別に設定
   する)。
