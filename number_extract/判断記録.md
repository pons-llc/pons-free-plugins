# 判断記録

`plugin_idea.md`の元メモ(「数字抽出プラグイン」の項)は数行と短く、実装時に暫定判断が必要だった事項を
記録する。作業は止めずに進めた。

## 1. 漢数字の変換を「位取り表記」と「読み上げ表記」の2パターンに自動判別した

- **問題**: 元メモは「漢数字を含むかどうかを選択する」とだけあり、`十二`(位取りで12)と`二〇二四`
  (読み上げで2024の年号)のように、漢数字には全く異なる2つの読み方があることには触れていなかった。
- **選んだ暫定対応**: 連続する漢数字の中に`十`・`百`・`千`・`万`のいずれかが含まれるかどうかで
  自動判別する方式にした。含まれていれば位取り変換(標準的な漢数字→算用数字アルゴリズム)、
  含まれていなければ1文字ずつを桁として連結する読み上げ変換とした。`〇`(ゼロ)は位取り表記では
  通常使われない(「百〇五」のような表記は一般的でない)ため、この判別ルールで実用上の曖昧さは
  小さいと判断した。
- **要ユーザー確認**: `〇`を含みつつ位取り表記も使う特殊なケース(例: `十〇`のような表記)が
  実運用で必要であれば、判別ロジックの見直しが必要。

## 2. `億`以上の位に対応しない

- **問題**: 元メモに上限の記載がなかった。
- **選んだ暫定対応**: `万`までの対応とし、`億`・`兆`は対応しない(`js/lib/kanji-number.js`の
  `parseSmall()`は`十`・`百`・`千`のみ、`kanjiToNumber()`で`万`を1回だけ分離する設計)。
  住所の番地・号・丁目や小さな年号の読み上げといった想定用途では、`億`単位の数字が現れることは
  実務上考えにくいと判断した。
- **要ユーザー確認**: `億`単位の漢数字を扱う用途があれば、`kanji-number.js`に`億`の分離処理を
  追加する拡張が必要(`万`の処理と同じパターンで再帰的に対応できる)。

## 3. 全角数字・漢数字の連続は算用数字の連続と別々にカウントする(混在時は分割される)

- **問題**: `"２丁目二番地"`のように全角数字と漢数字が1つの文字列に混在する場合の扱いが元メモに
  なかった。
- **選んだ暫定対応**: それぞれ独立した「連続する数字表記のまとまり」として抽出する。全角数字の連続
  (半角に変換後`\d+`にマッチ)と漢数字の連続(`〇一二三四五六七八九十百千万`の連続にマッチ)を
  別々の正規表現で走査し、出現位置順にマージする。地続きで両方の表記が混ざった1つの数(例:
  「２十３」のような表記)は想定していない。
- **要ユーザー確認**: 実際のデータにそのような混在表記が含まれる場合は、抽出結果が意図通りに
  ならない可能性がある。

## 4. `text_split`と共通する設計判断をそのまま踏襲

- 出力先フィールドが抽出結果より多い場合の切り捨て方針、設定行を複数持てる構成、
  発動タイミング(submitのみ)、出力先/元フィールドの`disabled`化の範囲は、`text_split/判断記録.md`の
  1番・3番・4番・5番と同じ考え方をそのまま踏襲した(理由の重複記載は省略する)。
- `wareki_date_format`由来の`eslint.config.mjs`(flat configファクトリー関数化対応)もそのまま
  コピーして使っている。

## 6. 発動タイミングを`submit`→`change`に、出力先フィールドの`disabled`化を撤廃(ユーザーレビューによる修正)

- **問題**: 初期実装では`text_split`と同じ「`submit`イベントで抽出→出力先フィールドは`disabled`で
  直接編集禁止」としていたが、ユーザーレビューで「挿入先の編集禁止はダメだ」との指摘を受けた。
- **修正内容**: `desktop.js`/`mobile.js`を以下の通り変更した。
  - 発動イベントを`app.record.create.submit`/`app.record.edit.submit`から、
    `app.record.create.change.<元フィールドコード>`/`app.record.edit.change.<元フィールドコード>`
    (元フィールドごとに動的に組み立てる、`wareki_date_format`と同じパターン)へ変更。`submit`イベントの
    ハンドラーは削除した。
  - 出力先フィールドの`disabled`化(`create.show`/`edit.show`ハンドラーとその中の
    `disableTargetFields()`)を完全に削除。出力先フィールドは常に編集可能なままにした。
  - 元フィールドを一覧のインライン編集で`disabled`にする処理(`app.record.index.edit.show`)は
    ユーザー指摘の対象外だったため変更していない。
- **結果として生まれる仕様**: `change`イベントで自動入力された出力先フィールドの値を、ユーザーが
  保存前に手で書き換えても、`submit`時に再上書きされることはない(そもそも`submit`のハンドラーが
  存在しないため)。「自動抽出はするが、最終的な値はユーザーが手直しできる」という運用になる。
- **適用範囲(ユーザー確認済み・確定)**: この修正は本プラグイン(`number_extract`)のみに適用する。
  同じ元メモの文言(「挿入先の編集を禁止。元フィールドは一覧からの編集を禁止。」)を持つ`text_split`は
  従来通り(`submit`発動+出力先`disabled`)のままとし、今後作成する同種プラグイン(`text_slice`等)も
  当面は元の仕様(`submit`発動+出力先`disabled`)で作る。プラグインごとに仕様が異なる状態になっている
  点に注意。
