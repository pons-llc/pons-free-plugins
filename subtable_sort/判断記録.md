# 判断記録

`plugin_idea.md`の元メモ(「サブテーブルソートプラグイン」の項)は数行と短く、実装時に暫定判断が
必要だった事項を記録する。作業は止めずに進めた。

## 1. `MANUAL`(任意実行)モードをPC専用とした

- **問題**: 元メモに「任意実行」のUI設置場所の指定がなかった。
- **選んだ暫定対応**: `status_arrow`/`confirm_modal`と同様、`kintone.app.record.getHeaderMenuSpaceElement()`
  (PC専用)にソート実行ボタンを描画することにした。`SUBMIT`モードはイベントハンドラーのみで完結する
  ためPC・モバイル両対応だが、`MANUAL`モードのボタンUIはPC専用となる。
- **要ユーザー確認**: モバイルでも任意実行ボタンが必要な場合、スペースフィールドを別途アンカーとして
  使う方式(`tab_layout`と同様の`getSpaceElement()`方式)への設計変更が必要。

## 2. ソート済フィールドの文字列(「済」/「未」)は固定とした

- **問題**: 元メモは「ソート済かどうかを挿入するフィールドを設定可能」「未にして」「済にする」と、
  フィールドの選択は設定可能だが、実際に挿入する文字列自体をカスタマイズ可能にするかは明記が
  なかった。
- **選んだ暫定対応**: 元メモに明記された「済」「未」の2値をそのまま固定文字列として使うことにした
  (`js/lib/flag-values.js`)。カスタマイズ可能にすると設定項目が増えるだけで、元メモの具体的な
  文言を再現するだけなら固定で十分と判断した。
- **要ユーザー確認**: 「対応済」「未対応」のような別の文言にしたい場合は、`flag-values.js`の定数を
  変更するか、設定画面にテキスト入力を追加する拡張が必要。

## 3. `confirm_modal`との連携は「同じアプリに両方インストールする」レベルに留めた

- **問題**: 元メモの「モーダル確認と組み合わせることでいい感じ」が、技術的な連携(例:
  ソート実行前に本プラグインが`confirm_modal`のAPIを呼ぶ)を意味するのか、単に「同じアプリに
  両方入れておくと運用上便利」という程度の言及なのかが曖昧だった。
- **選んだ暫定対応**: 技術的な連携は行わない。`confirm_modal`は`app.record.edit.submit`等の
  汎用イベントに対して確認ダイアログを出すプラグインであり、`SUBMIT`モードのソートと組み合わせれば
  「保存前に確認ダイアログが出て、OKを押すと保存されソートも実行される」という体験が
  **プラグイン間の連携コードなしに**自然に実現できる(`confirm_modal`が保存イベントをフックし、
  本プラグインも同じ保存イベントをフックしているため、両方インストールするだけで元メモの
  「組み合わせることでいい感じ」が成立する)。CLAUDE.mdの「各プラグインが独立」という方針にも
  合致する。
- **要ユーザー確認**: 特になし(方針通りの独立設計)。

## 4. ソートキーの値の型を`STRING`/`NUMBER`の2種類とした

- **問題**: 元メモに「複数からむ条件」とあるが、型ごとの比較方法(数値/文字列/日付)の指定は
  なかった。
- **選んだ暫定対応**: `STRING`(文字列としての辞書式比較)と`NUMBER`(数値としての比較)の2種類に
  絞った。日付・日時・時刻型のkintoneフィールド値はISO8601形式の文字列で、辞書式順序が時系列順序と
  一致するため、`STRING`型として扱えば追加の型変換なしに正しくソートできる(`wareki_date_format`の
  タイムゾーン変換とは異なり、単純な大小比較だけなのでタイムゾーンの影響を受けない)。
- **要ユーザー確認**: 特になし(実用上、日付列は`STRING`指定で正しく動作する)。
