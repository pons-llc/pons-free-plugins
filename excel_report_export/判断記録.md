# 判断記録

plugin_idea_plan.md に明記がなく、実装中に新たに判断が必要になった事項を記録する。

## 1. テンプレート参照先レコード・フィールドが削除/変更された場合のエラーハンドリング(計画書で既知の未決事項)

plugin_idea_plan.mdの「複雑性・リスク・未決事項」で「新たな未決事項」として挙がっていた項目。

- **暫定対応(確定)**: `js/template-source.js`の`fetchTemplateRecord` / `extractFirstFileKey` /
  `downloadFileArrayBuffer`は、対象アプリ・レコードが存在しない、フィールドコードが変更/削除されている、
  添付ファイルフィールドにファイルが1つもない、閲覧権限がない、などのケースで、原因を特定できる
  日本語メッセージ付きの例外を投げる。呼び出し元(`single-export.js` / `bulk-export.js`)はこれを
  `catch`し、`alert()`でエラーメッセージを表示した上で処理を中断する(部分的な出力は行わない)。
- 今後の拡張候補: 設定画面に「テンプレート接続確認」ボタンを設け、保存前にテンプレートが取得できるかを
  検証できるようにする(未実装。今回のタスクスコープ外)。

## 2. `GET /k/v1/file.json` は `kintone.api()` を使えない(公式ドキュメントの制限事項)

CLAUDE.mdの開発方針3は「REST APIはJavaScript APIで実現できない場合のみ使い、その際も
kintone自身への呼び出しに限り`kintone.api()`(内部向けラッパー)を使用する。生の`fetch`/`XHR`で
直接URLを組み立てない」としているが、kintoneドキュメントMCPで
`kintone REST APIリクエストを送信する`(`kintone.api()`)のページを確認したところ、
「制限事項」に**ファイルをダウンロードするAPI(`GET /k/v1/file.json`)は`kintone.api()`では
実行できない**と明記されており、公式ドキュメントのサンプルコード自体が`fetch()`を使っている。

- **対応(確定)**: `js/template-source.js`の`downloadFileArrayBuffer`は`fetch()`を使う。
  ただし以下の点でCLAUDE.mdの意図(生URLの組み立て禁止・kintone自身への呼び出しに限定)を維持している。
  - URLは`kintone.api.url('/k/v1/file.json', true)`(kintone JavaScript API)で組み立てており、
    ドメインをハードコードしていない。
  - `fetch()`の宛先は常にkintone自身の`/k/v1/file.json`のみで、外部サーバーへの通信は行わない。
  - リクエストヘッダーはドキュメントのサンプルコードどおり`X-Requested-With: XMLHttpRequest`のみを付与し、
    認証情報(Cookie)はブラウザが同一オリジンへ自動付与するセッションCookieに委ねる(コード上で
    認証情報を扱わない)。
- この例外は`security-checklist.md`にも記載している。

## 3. `kintone.app.getQuery()` は実在せず、正しくは `kintone.app.getQueryCondition()`

plugin_idea_plan.mdの「6. 神エクセル帳票印刷プラグイン」節、および共通前提の記載は
「`kintone.app.getQuery()`」という表記だったが、kintoneドキュメントMCPで確認したところ、
現在のレコード一覧の絞り込み条件(クエリ文字列)を取得するAPIの正式名称は
**`kintone.app.getQueryCondition()`**(モバイルは`kintone.mobile.app.getQueryCondition()`)である。
`plugin_idea_plan.md`冒頭の「共通の前提・訂正事項」表にはこの訂正が載っていなかったため、
本プラグインの実装(`js/bulk-export.js`)では実際のAPI名`kintone.app.getQueryCondition()`を使用した。
他のプラグイン(構想3・4等)でも同様の表記揺れがあれば、実装時に同様の確認・訂正が必要になる。

## 4. モバイル非対応(PCのみ)

kintoneドキュメントMCPで確認したところ、1件ダウンロードのボタン設置に使う
`kintone.app.record.getHeaderMenuSpaceElement()`は「利用できる画面: PC」のみで、モバイルには
存在しない(モバイルはヘッダー直下の別APIになるが、レコード詳細画面のメニュー領域という
同等の設置場所ではない)。また一括ダウンロード(zip生成)はブラウザメインスレッドでの同期処理
(fflateの`zipSync`)や複数xlsxの同時メモリ展開を伴い、モバイル端末での動作は現実的でないと判断した。

- **対応(確定)**: 本プラグインは`manifest.json`に`mobile`ブロックを設けず、PC専用とする。
  `idea.md`にも明記済み。将来モバイル対応が必要になった場合は、1件ダウンロードのみ
  (一括ダウンロードは対象外)として別途設計する。

## 5. 外部ライブラリのブラウザ向け配布物を「ビルド時コピー」する方式(バンドラー不使用の維持)

plugin_idea_plan.mdは「ライブラリはビルド時にバンドルし、CDN読み込みや外部変換サービスは使わない」と
述べているが、CLAUDE.mdの基本方針は「バンドラーを意図的に導入しない」である。この2つを両立するため、
webpack/rollup/esbuild等のバンドラーは導入せず、ExcelJS・fflateがそれぞれnpmパッケージ内に同梱している
ブラウザ向けビルド済みファイル(`exceljs/dist/exceljs.min.js`, `fflate/umd/index.js`)を、
`pnpm run build`の一部(`scripts/copy-vendor.js`)で`js/vendor/`へコピーし、`manifest.json`から
通常のローカルjsファイルとして読み込む方式にした。

- `js/vendor/`配下は`dist/`と同様に`.gitignore`対象(生成物として扱う)。`pnpm install`後に
  `pnpm run build`(内部で`pnpm run vendor`を実行)すれば再生成される。
- ESLintの対象からも`js/vendor/**`を除外している(圧縮された巨大ファイルをパースしようとして
  `eslint .`が長時間ハングする事象が実際に発生したため、`eslint.config.mjs`の`ignores`に追加した)。

## 6. `pnpm audit`での脆弱性是正: `uuid`のバージョン固定は`pnpm-workspace.yaml`の`overrides`で行う

`package.json`の`pnpm.overrides`フィールドは、インストール時に
`[WARN] The "pnpm" field in package.json is no longer read by pnpm.`という警告が出て**無視される**
(pnpm v10以降、`pnpm-workspace.yaml`の`overrides`フィールドへ移動済み)ことが判明した。
そのため、ExcelJSの推移的依存`uuid`(`^8.3.0`、既知の脆弱性GHSA-w5hq-g745-h8pqあり)を
`11.1.1`に固定する設定は`pnpm-workspace.yaml`側に記載している。`pnpm install`後
`pnpm ls uuid --depth 10`で実際に`11.1.1`が解決されていること、`pnpm audit`が0件になることを確認済み
(security-checklist.md参照)。

## 7. セル書き込み失敗時の扱い: 部分失敗を許容し、警告を出しつつファイルは生成する

結合セルの非マスターセルへ書き込もうとした場合など、テンプレートのレイアウトとマッピング設定の
不整合によりExcelJSがエラーを投げるセルが発生し得る(神エクセル特有の複雑なテンプレートで
起こりやすいリスクとしてplugin_idea_plan.mdも指摘している)。

- **対応(確定)**: `js/workbook-builder.js`の`applyWrites`は、書き込み命令1件ごとに`try/catch`し、
  失敗した命令だけを警告メッセージとして集約する。1つのセルの失敗で処理全体を中断せず、
  書き込める他のセル・シートは書き込んだ上でファイルを生成・ダウンロードし、最後に警告をまとめて
  `alert()`で表示する。サブテーブルの行数が`maxRows`を超えた場合も同様に、超過分を書き込まずに
  警告として表示する(サイレントな切り捨てはしない)。

## 8. 一括ダウンロードの上限チェックは軽量な件数取得を先に行う

`js/record-fetcher.js`の`fetchTotalCount`は、`GET /k/v1/records.json`に`limit 1` +
`totalCount: true` + `fields: ['$id']`を指定して呼び出し、実際のレコード内容を取得せずに
対象件数のみを取得する。これにより、上限超過時(`js/lib/bulk-limit.js`の`checkBulkLimit`が
`allowed: false`を返す場合)に無駄な全件取得・Excel生成処理を行わずに済む。

## 9. 一括ダウンロードの確認ダイアログには`kintone.showConfirmDialog()`を使う

CLAUDE.md/plugin_idea_plan.mdの「共通の前提・訂正事項」表に記載のとおり、構想メモの
`kintone.app.showConfirmDialog()`は実在せず、正しくは`kintone.showConfirmDialog(config)`
(Promiseが`OK`/`CANCEL`/`CLOSE`で解決)である。`js/bulk-export.js`はこの表の訂正に従い、
一括ダウンロード実行前の確認に`kintone.showConfirmDialog()`を使用している。

## 10. 一括ダウンロード時のテンプレートArrayBufferの使い回し

`js/bulk-export.js`は、テンプレートファイルを1回だけ取得し、レコードごとに
`templateArrayBuffer.slice(0)`で独立したコピーを`WorkbookBuilder.buildWorkbook`へ渡している。
ExcelJSの`workbook.xlsx.load()`は読み込んだバッファの中身を内部状態として保持するため、
同一のArrayBufferをレコード間で使い回すと予期しない副作用が起きる可能性があると判断し、
安全側に倒してコピーする方式にした(メモリ使用量は増えるが、一括ダウンロードは上限件数で
歯止めをかけているため許容範囲と判断)。

## 11. 出力ファイル名の重複解消(zipエントリ名の衝突回避)

一括ダウンロードで複数レコードのファイル名テンプレート出力が同名になるケース(例: 同じ顧客名の
レコードが複数件ある)を想定し、`js/lib/filename-template.js`の`dedupeFileNames`で
`(2)`, `(3)`のような連番を付与して衝突を解消する。zip内で同名エントリが上書き・混在するリスクを防ぐ。
