### 監査記録(`pnpm audit`, 2026-07-09時点)

```
src@0.1.0
├─ exceljs 4.4.0
│  └── uuid 11.1.1  (pnpm-workspace.yamlのoverridesでuuid ^8.3.0 → 11.1.1に固定)
└─ fflate 0.8.3
```

- `pnpm audit`実行結果: **2件検出**(いずれも本プラグインの実行コードには含まれない、
  ビルド用devDependency `@kintone/cli` の推移的依存)。
  - `form-data`(`@kintone/cli > @kintone/rest-api-client > axios/form-data`, High,
    GHSA-hmw2-7cc7-3qxx)
  - `qs`(`@kintone/cli > @kintone/rest-api-client > qs`, Moderate, GHSA-q8mj-m7cp-5q26)
  - この2件は`fiscal_year_numbering`など既存プラグインでも同一の`pnpm audit`結果になることを確認済み
    (`@kintone/cli`自体のロックファイル起因で、本プラグイン固有の追加ではない)。`@kintone/cli`は
    ローカルのビルド・パッケージング専用のdevDependencyであり、`pnpm run build`が生成する
    `dist/plugin.zip`(実行コード)には含まれない。box_gdrive_iframe/security-checklist.mdの
    「N/A — 実行コードに外部パッケージを持たない」と同じ整理。
- **exceljs / fflate 自体(実行コードに含まれる2ライブラリ)に起因する脆弱性は0件**。
  - exceljsの推移的依存`uuid`(既定`^8.3.0`)は既知の脆弱性(GHSA-w5hq-g745-h8pq、
    「uuid v3/v5/v6で呼び出し元がbufを渡した場合のバッファ境界チェック漏れ」)が指摘されていたが、
    exceljs内部での実際の使用箇所(`lib/xlsx/xform/sheet/cf-ext/cf-rule-ext-xform.js`)は
    `uuid`の`v4`(引数なしのランダムUUID生成)のみで、脆弱性の影響経路(v3/v5/v6へのbuf指定)には
    該当しないと判断した。とはいえ多層防御として`pnpm-workspace.yaml`の`overrides`で
    `uuid: 11.1.1`(修正済みバージョン)に固定し、`pnpm audit`上の指摘自体も解消した。

### 更新・代替手順(重大な脆弱性が報告された場合)

1. `pnpm audit`または GitHub Advisory Database でexceljs/fflateに新たな脆弱性が報告されていないか
   定期的に確認する(バージョンを上げるたびに必須、それ以外も年1回程度を目安に確認する)。
2. 修正版が存在する場合は、`package.json`のバージョンを完全固定のまま新バージョンへ更新し、
   `pnpm install` → `pnpm audit` → idea.mdのスパイク手順(結合セル・罫線・列幅を含むテンプレートの
   読み込み→書き込み→再読込)を再実行して書式保持に回帰がないことを確認してから反映する。
3. 修正版が存在しない、またはメンテナンスが停止した場合:
   - Excel操作ライブラリの代替候補: 罫線・結合セルの保持を最優先し、同種の要件を満たす別ライブラリ
     (例: 当時のSheetJS Pro版や他のOOXML操作ライブラリ)を同じスパイク手順で再評価する。
   - zip操作ライブラリの代替候補: `jszip`等、依存0件に近い他のzipライブラリを再評価する。
   いずれの場合も、代替後は本チェックリストの「監査記録」を更新すること。
4. 本プラグインの実行コードは外部サーバーと通信しないため、ライブラリ自体に深刻な脆弱性があっても
   「悪意あるスクリプトの動的取得」のような即時侵害経路は存在しない(CDN不使用・ビルド時コピー方式、
   判断記録.md参照)。ただし旧バージョンのまま放置しないよう、上記の定期確認は継続する。

## XSS・CSSインジェクション対策

- [x] レコード値やユーザー入力を`innerHTML`で出力していない。設定画面(`js/config.js`)は
      `textContent`(オプションのラベル生成)と`value`プロパティのみを使用し、`innerHTML = ''`は
      リストの再描画時のクリア用途のみ(固定文字列の代入であり、外部入力を差し込んでいない)。
- [x] Excel出力そのものはDOMへHTMLとして挿入するものではなく、Blobとしてダウンロードされるファイルの
      ため、XSSの一般的な経路(innerHTML注入)には該当しない。ただし添付ファイル型テンプレートは
      「任意のアプリ」から取得するため、次項の「テンプレートの信頼性」を参照。
- [x] `a[download]`要素の`href`には、本プラグイン内で生成した`Blob`のオブジェクトURL
      (`blob:`から始まるURL)のみを設定しており、外部からの入力(レコード値等)をURLとして
      組み立てていない(`js/download-file.js`)。secureCodingGuideline.mdの「出力するURLは
      http(s)のみ」より厳格に、外部由来の値を一切URLに使わない設計。
- [x] RICH_TEXTフィールドの値をセルに書き込む際は、HTMLタグを除去したプレーンテキストに変換してから
      書き込んでいる(`js/lib/field-value-format.js`の`stripHtmlTags`)。Excel側にHTMLとして
      解釈される余地はそもそも存在しないが、タグ文字列がそのままセルに表示される見た目の悪化は防止している。

## テンプレートの信頼性(添付ファイルフィールド参照方式に伴うリスク)

- [ ] **管理者が任意のアプリ・レコード・添付ファイルフィールドをテンプレート参照先として設定できる**
      ため、悪意または誤りのある管理者が意図しないアプリのファイルを参照させることが理論上可能。
      ただしこの設定自体はプラグイン設定(`kintone.plugin.app.setConfig()`)であり、アプリ管理者
      以上の権限を持つユーザーのみが変更できる(kintoneのプラグイン設定変更権限に準拠)。
      一般利用者が改ざんできる経路ではないため、追加の対策は行っていない。
- [x] テンプレートファイル自体はExcelJSで「読み込んで値を書き込み、xlsxとして書き出す」処理のみを
      行い、テンプレートに含まれるマクロ(VBA)やOLEオブジェクトを実行することはない
      (ExcelJSはブラウザ上で完結するパーサーであり、Excelアプリケーションのようにマクロを
      実行する機能を持たない)。

## テンプレート参照先への権限がないユーザーの挙動(secureCodingGuideline.md準拠の確認事項)

- [x] テンプレート取得(`GET /k/v1/record.json`, `GET /k/v1/file.json`)は、ダウンロードを実行した
      **ユーザー自身のセッション**(`kintone.api()`および同一オリジンの`fetch()`、いずれもCookie
      ベースのセッション認証)で行われる。したがって、テンプレート参照先アプリ・レコード・
      フィールドへの閲覧権限がないユーザーが実行した場合、kintone側の通常の権限チェックにより
      403相当のエラーが返り、本プラグインがその権限チェックをバイパスすることはない
      (「必要なアクセス権」はkintoneドキュメント記載のとおり、アプリのレコード閲覧権限・
      対象レコードの閲覧権限・対象フィールドの閲覧権限)。
- [x] 権限エラー発生時は`js/template-source.js`の例外がそのまま`single-export.js`/
      `bulk-export.js`の`catch`に伝播し、`alert()`でエラーメッセージを表示して処理を中断する
      (暫定対応。判断記録.md「1. テンプレート参照先レコード・フィールドが削除/変更された場合の
      エラーハンドリング」参照)。テンプレートの内容やエラーの詳細を必要以上に開示するメッセージには
      していない(kintone側のエラーレスポンスをそのまま転記せず、「取得に失敗した」旨の日本語文言に
      要約している)。

## ダウンロードしたファイルの命名(インジェクション・パストラバーサル対策)

- [x] 出力ファイル名は`js/lib/filename-template.js`の`buildFileName`で、Windows/macOSで使用できない
      文字(`\ / : * ? " < > |`)および制御文字を除去し、前後の空白・ドットを取り除いた上で
      `.xlsx`拡張子を付与する(パス区切り文字`/`\`を除去しているため、`a[download]`属性値による
      意図しないパス指定を防止している)。テストは`__tests__/filename-template.test.js`参照。
- [x] サニタイズの結果が空文字列になる場合は固定のフォールバック名(`record`)を使用し、
      空のファイル名や`.`のみのファイル名を生成しない。
- [x] 長すぎるファイル名は200文字以内に切り詰める(OSのパス長制限を考慮)。
- [x] 一括ダウンロード時、複数レコードが同名になった場合は`dedupeFileNames`で`(2)`,`(3)`のような
      連番を付与し、zip内でのファイル名衝突(上書き)を防止している。

## 通信・認証情報の取り扱い

- [x] kintone以外のサーバーへの外部通信を一切行わない(ExcelJS/fflateはビルド時にコピーした
      ローカルファイルとして読み込み、CDN読み込み・外部変換サービスへの送信は行わない)。
- [x] 認証情報(APIトークン等)を一切扱わない。すべてのkintone API呼び出しは
      `kintone.api()`(内部向けラッパー、実行ユーザー自身のセッション)、または`GET /k/v1/file.json`
      のみ例外的に`fetch()`(同一オリジン、Cookieベースのセッション認証を自動付与)で行う。
- [x] プラグイン設定(`kintone.plugin.app.setConfig()`)に保存しているのはテンプレート参照先
      (アプリID・レコードID・フィールドコード)、セルマッピング、サブテーブル設定、ファイル名
      テンプレート、上限件数のみで、認証情報・機密情報は含まれない。

## リダイレクト・URL生成

- [x] `window.location.href`に渡す値は`kintone.app.getId()`など信頼できる内部値のみ(`js/config.js`)。
- [x] REST APIのURLは`kintone.api.url()`(secureCodingGuideline.mdの「URLの取得」に準拠)で組み立てて
      おり、ドメインを直接文字列結合していない。

## kintone REST API利用上の注意点

- [x] 一括ダウンロードのレコード取得は`for`ループによる逐次ページング(`js/record-fetcher.js`)であり、
      並列で大量のリクエストを同時実行していない(secureCodingGuideline.mdの「短時間で大量の
      リクエスト送信を避ける」「並列で実行するのをなるべく避ける」に準拠)。
- [x] 一括ダウンロードは設定画面で必須の上限件数(既定100件)でリクエスト数に歯止めをかけている。

## 個別確認事項(利用ユーザーへ委ねる項目・今回のタスクスコープ外)

- 複雑なテンプレート(多段結合セル・図形・印刷設定)がExcelJSの読み書きで崩れないかは、
  実際の利用者テンプレートを用いたPuppeteer e2eテスト(CLAUDE.md項目6、本タスクのスコープ外)で
  別途検証する。
- `kintone.showConfirmDialog()` / `kintone.showLoading()`の実際の見え方・タイミングは
  Puppeteerでの実環境テストで検証する(本タスクのスコープ外)。
- 各ブラウザ(Chrome/Edge/Safari等)でのBlobダウンロード・zipダウンロードの挙動差異は、
  個別の不具合として利用ユーザーからのGitHub Issue報告で対応する。

問題があれば、公開サイトのリポジトリのGitHub Issueで報告してもらい対応する。
