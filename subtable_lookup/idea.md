# サブテーブルルックアッププラグイン

## 機能

サブテーブルの中から条件に一致する1行を探し、その行の複数フィールド(列)の値を、テーブル外の複数フィールドへコピーする。
点検履歴などをサブテーブルで管理し、「最新の点検結果」「最終点検日」「特定条件に合う行の値」を
テーブル外のフィールドへ自動反映する用途を想定している。

- ルックアップ設定(以下「設定行」)を複数持てる。1つのサブテーブルに対して複数の設定行を作ることも、
  複数の異なるサブテーブルに対してそれぞれ設定することもできる。
- 設定行ごとに、対象サブテーブル・検索モード・(モードに応じた)検索対象列・一致させる値・検索方向・
  フィールドマッピング(サブテーブル列→出力先フィールドの複数ペア)を持つ。

## 検索モード(確定・6種類)

| モード | 検索対象列 | 一致させる値 | 動作 |
| :-- | :-- | :-- | :-- |
| `PARTIAL_MATCH`(部分一致) | 必須(型不問、文字列化して比較) | 必須 | 検索対象列の値に指定文字列が含まれる行を探す |
| `EXACT_MATCH`(完全一致) | 必須(型不問、文字列化して比較) | 必須 | 検索対象列の値が指定文字列と完全一致する行を探す |
| `LATEST`(最新) | 必須(DATE/DATETIME/TIME型) | 不要 | 検索対象列の値が最大(最も新しい)行を探す |
| `OLDEST`(最古) | 必須(DATE/DATETIME/TIME型) | 不要 | 検索対象列の値が最小(最も古い)行を探す |
| `TOP_ROW`(一番上の行) | 不要 | 不要 | サブテーブルの先頭行を無条件に採用する |
| `BOTTOM_ROW`(一番下の行) | 不要 | 不要 | サブテーブルの末尾行を無条件に採用する |

## 検索方向(確定)

`TOP_TO_BOTTOM`(上から)/ `BOTTOM_TO_TOP`(下から)の2種類。役割はモードによって異なる。

- `PARTIAL_MATCH` / `EXACT_MATCH`: 複数行が条件に一致したとき、上から探して最初に見つかった行を採用するか、
  下から探して最初に見つかった行(=テーブル順で最後に一致した行)を採用するかを決める。
- `LATEST` / `OLDEST`: 検索対象列の値が同値の行が複数あるとき、上から数えて最初の行を採用するか、
  下から数えて最初の行(=テーブル順で最後の同値行)を採用するかのタイブレークに使う。
- `TOP_ROW` / `BOTTOM_ROW`: モード自体が行の位置を指定しているため、検索方向は無視する(設定画面では
  非表示にする)。

日付・日時・時刻の値(`DATE`は`"YYYY-MM-DD"`、`DATETIME`はISO8601のUTC文字列、`TIME`は`"HH:MM:SS"`)は、
いずれもゼロ埋め・辞書式順序が時系列順序と一致するフォーマットなので、`Date`オブジェクトへの変換なしに
**文字列の大小比較のみ**で新旧を判定する(`js/lib/row-finder.js`)。

## 一致しなかった場合の挙動(確定)

条件に一致する行が1件もない場合、その設定行に紐づくすべての出力先フィールドを**空文字列でクリアする**。
「前回保存時の値が古いまま残る」事故を避けるため、常に上書きする方針にした(`wareki_date_format`の
「出力先フィールドは常に上書きする」方針を踏襲)。

## フィールドマッピング

設定行ごとに、サブテーブル列コード→出力先フィールドコードのペアを複数持てる。出力先フィールドコードは
プラグイン全体(すべての設定行を通じて)で重複を禁止する(重複すると2つの設定行の結果が同じフィールドを
取り合い、後勝ちの挙動が予測しづらくなるため。`wareki_date_format`と同じ理由による判断)。

型の互換性チェックは行わない(サブテーブル列の値をそのまま文字列としてコピーする、シンプルな代入)。
出力先フィールドの型とサブテーブル列の型が一致しない場合の挙動はkintone側の入力チェックに委ねる。

## 発動タイミング(確定・明記)

**`app.record.create.submit` / `app.record.edit.submit` イベントでのみ動作する。**
`change`イベントでの即時プレビューや`show`イベントでの先行計算は行わない(スコープ外、`判断記録.md`参照)。

## 設定画面

`kintone.plugin.app.setConfig()`にのみ保存する。

- 設定行を追加・削除できる。設定行ごとに:
  - 対象サブテーブル(`kintone.app.getFormFields()`で`SUBTABLE`型のフィールドを列挙)
  - 検索モード(上記6種類)
  - 検索対象列(対象サブテーブル選択後、そのサブテーブル内のフィールドを動的に列挙。`LATEST`/`OLDEST`選択時は
    `DATE`/`DATETIME`/`TIME`型のみに絞り込む。`TOP_ROW`/`BOTTOM_ROW`選択時は非表示)
  - 一致させる値(`PARTIAL_MATCH`/`EXACT_MATCH`選択時のみ表示、テキスト入力)
  - 検索方向(`TOP_ROW`/`BOTTOM_ROW`以外で表示)
  - フィールドマッピング(サブテーブル列→出力先フィールドの複数ペアを追加・削除できる)
- 保存時に`js/lib/config-validation.js`でチェックし、不正な設定では保存させない。

## TDD

`src/js/lib/`配下の純粋ロジックはJestでユニットテストする(`pnpm test`)。

- `row-finder.js` — サブテーブルのvalue配列+設定行から、条件に一致する1行(またはnull)を返す
- `row-mapper.js` — 一致行(またはnull)+フィールドマッピングから、出力先フィールドの値オブジェクトを組み立てる
  (一致行がnullのときは全出力先フィールドを空文字列にする)
- `config-store.js` — `kintone.plugin.app.getConfig()`/`setConfig()`のペイロードの読み書きとデフォルト値
- `config-validation.js` — 設定(設定行の配列)のバリデーション

kintone依存のグルーコード(`desktop.js`/`mobile.js`/`config.js`)は今回のタスクスコープでは実環境テスト
(Puppeteer)を行わない(別タスク、e2eは後回し)。

## 実装

kintoneドキュメントMCPを参照しながら実装した(SUBTABLE型フィールドの値構造 `{ value: [{ id, value: { colCode: { type, value } } }] }`、
`app.record.create.submit`/`app.record.edit.submit`イベント、`kintone.app.getFormFields()`でのサブテーブル内フィールド列挙)。
