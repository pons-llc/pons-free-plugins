# ガントチャートプラグイン

## 機能概要

レコード一覧画面(**表形式**, `viewType: 'list'`)のヘッダー領域にガントチャートを表示する、**PC専用・表示専用**のプラグイン。

- 対象は通常の一覧(表形式)。カスタマイズビューではなく、`kintone.app.getHeaderMenuSpaceElement()`(集計アイコンの右側の要素、レコード一覧画面全般で利用可)にガントチャートのブロックを独立して挿入する。一覧本体の表(行)には手を入れない。
- `app.record.index.show` イベントの `event.viewType === 'list'` のときのみ対象と判定し、`event.viewId` で一覧ごとの表示設定を引き当てる。一覧ごとに表示設定を切り替えられる。
- 開始日・終了日フィールド(日付/日時)を指定してバーを描画する。
  - 終了日が未設定の行は、開始日と同じ1日幅のバーとして表示する。
  - 開始日が未設定の行は「未仕訳」グループとして最下部にまとめて表示する(バーは描画しない)。
- グループ分け: 作業者(ユーザー選択)/組織/グループ選択/ドロップダウン/ラジオボタンの各フィールドでグループ分けできる。管理者は「一覧画面で切り替え可能なグループ分けフィールド」を複数設定でき、そのうち1つを既定のグループ分けとして指定する。一覧画面ではツールバーのドロップダウンからグループ分けを切り替えられる(ページ内の一時的な状態で、保存はされない)。
- バー内(および行ラベル)に表示する項目を複数フィールド選択できる。色分けに使うフィールドも設定でき、値ごとに固定パレットから決定的に色を割り当てる。
- 時間軸のスケールを日/週/月で切り替えられる(ツールバーのボタン、ページ内の一時的な状態)。
- 全件取得ボタン(設定で有効/無効を切り替え可能)。押すと、現在の絞り込み条件を維持したまま `$id` 昇順ページングでアプリの全レコード(上限件数まで)を取得し、取得した全件でガントチャートを再描画する。
- 描画性能のため、全件取得時の件数上限を設定できる(既定値2000件)。上限に達した場合は打ち切り、その旨をツールバーに表示する。

## 対象一覧(view)の作成・管理

- 対象の一覧(表形式)自体は**管理者が事前に手動で作成する**。プラグインによる一覧の自動作成(`PUT /k/v1/app/views.json`)は行わない。
- **重要な制約**: `GET /k/v1/app/views.json` のレスポンスには、ビルトインの「すべて」ビュー(ユーザーが作成していないデフォルトの一覧)が**含まれない**。そのため設定画面では「すべて(デフォルト)」を固定の選択肢として別枠で用意し、`event.viewId` がAPI取得済み一覧のどれにも一致しない場合はこの「すべて」設定にフォールバックする(`js/lib/paging-query.js` の `resolveViewConfig`)。
- 一覧が後から削除・変更された場合、設定画面に残った古い一覧設定(viewIdが浮いた状態)は実害はない(該当するviewIdの一覧が二度と表示されなければ単に使われなくなるだけ)が、設定画面上に不要なエントリとして残り続ける。手動で削除する運用を想定し、自動クリーンアップは行わない(初期スコープ外)。

## 全件取得の実装方針

- **`offset`・カーソルAPIは使わない**。`kintone.app.getQuery()` ではなく `kintone.app.getQueryCondition()` で現在の絞り込み条件(query部分のみ、order by/limit/offsetを含まない)を取得し、`$id` 昇順 + `limit 500` で `GET /k/v1/records.json` をページングする(1回目は絞り込み条件のみ、2回目以降は `$id > 直前取得分の最大$id` を条件に追加)。offset上限1万件や生成コストの問題を回避できる、kintone推奨のページングパターン。
- クエリ合成ロジックは `js/lib/paging-query.js` に切り出し、Jestでユニットテストしている。
- REST APIはこの全件取得と、対象一覧の列挙(`GET /k/v1/app/views.json`、JavaScript APIに相当機能がないため)にのみ使用する。それ以外(フォーム項目の取得等)はすべて `kintone.app.getFormFields()` 等のJavaScript APIを使う(CLAUDE.md方針3)。

## 描画方式

- 外部ライブラリは一切使用せず、DOM+CSSで自前実装する(SVGは不使用。理由: DOM要素ベースの方がラベルのテキスト折り返し・ツールチップ・イベントハンドリングを素朴なDOM APIだけで組み立てやすく、`textContent` によるXSS対策も一貫させやすいため)。
- 時間軸は「1日あたりのピクセル数」を基準に、日/週/月のスケールごとに換算する(月スケールは30日換算の近似幅を採用し、グリッド線のみ実際のカレンダー月初日に合わせて描画する簡易実装。正確な月幅が必要になった場合は将来の課題とする)。
- レコード値に由来する文字列は、行ラベル・バー内ラベル・グループ見出しを含め、すべて `textContent` で挿入する(`innerHTML`/`insertAdjacentHTML` は不使用)。詳細は `security-checklist.md` を参照。

## モバイル対応

`kintone.app.getHeaderMenuSpaceElement()` はPC専用のため、本フェーズではモバイル対応を**行わない**(`mobile.app.record.index.show` には未対応)。将来モバイル対応する場合は、モバイル一覧画面向けの別の要素取得APIを別途調査し、モバイル用の描画(縦スクロール前提のレイアウトなど)を再設計する必要がある。

## 既知の制約・将来課題

- **バーのドラッグによる期間編集はスコープ外**。表示専用のプラグインとして割り切っている。
- **仮想スクロールは未実装**。全件取得時に大量件数(上限に達しない範囲でも数百〜2000件)を描画すると、DOM要素数に比例して描画・スクロール性能が低下する可能性がある。将来的に、表示領域外の行を描画しない仮想スクロールの導入を検討する。
- 月スケールの日数換算は30日固定の近似(上記「描画方式」参照)。
- 一覧の削除に伴う設定の自動クリーンアップは行わない。
- グループ分け・スケール単位の選択はページ内の一時的な状態であり、リロードすると設定画面の既定値に戻る(意図的な設計。CLAUDE.mdの「認証情報や個人設定をプラグイン設定に保存しない」方針とは別軸だが、UI状態の永続化自体を今回のスコープに含めていない)。

## 設定画面

対象一覧(view)ごとに、以下を設定する(`kintone.plugin.app.setConfig()` にのみ保存する)。

- 対象一覧の選択(「すべて(デフォルト)」を含む。REST API `GET /k/v1/app/views.json` の `type: LIST` の一覧から選ぶ)
- 開始日フィールド(必須、DATE/DATETIME)
- 終了日フィールド(任意、DATE/DATETIME。未設定なら開始日と同じ1日幅で表示)
- バー内に表示する項目(複数選択可、テーブル外のフィールドから選択)
- 色分けに使う項目(任意、ドロップダウン/ラジオボタン/チェックボックス/複数選択/ユーザー選択/組織/グループ選択/ステータスから選択)
- 既定のグループ分けフィールド + 一覧画面で切り替え可能なグループ分けフィールド(複数選択可、作業者/組織/グループ選択/ドロップダウン/ラジオボタンから選択)
- 全件取得ボタンの有効/無効
- 全件取得時の件数上限(既定値2000件)

## TDD

`src/js/lib/` 配下の以下の純粋ロジックをJestでユニットテストしている(`pnpm test`)。

- `record-model.js`: レコード配列→ガント描画用データモデルへの変換(開始日/終了日の欠損処理、未仕訳判定)
- `grouping.js`: グループ分けロジック(指定フィールドでのグルーピング、未仕訳グループの最下部固定、既定/利用可能グループ分けフィールドの検証)
- `time-scale.js`: 時間軸スケール計算(日付範囲→表示グリッド座標への変換。日/週/月スケール)
- `color-assignment.js`: バー色分けロジック(色分けフィールド値→色の決定的な割り当て)
- `paging-query.js`: `$id` 昇順ページングのクエリ合成ロジック、および対象一覧(view)のフォールバック判定ロジック(`event.viewId` がAPI一覧に存在しない場合の「すべて」判定)
- `config-store.js`: プラグイン設定の読み書き(`getConfig()` が `null` を返す場合の既定値フォールバックを含む)

kintone依存のグルーコード(`js/full-fetch.js`, `js/gantt-render.js`, `js/desktop.js`, `js/config.js`)は、CLAUDE.md項目6のPuppeteerによる実環境テストで別途検証する(**今回のタスクスコープ外**。`pnpm run upload` は未実行、`.env` の実環境kintoneドメインへの接続もまだ行っていない)。

## 実装で確認したkintone API仕様(kintoneドキュメントMCPで確認済み)

- `app.record.index.show`: `event.viewId`(一覧ID)・`event.viewType`(`list`/`calendar`/`custom`)・`event.records`(表形式のときは現在ページのレコード配列)を持つ。
- `kintone.app.getHeaderMenuSpaceElement()`(PC専用): 「集計アイコンの右側」の要素を取得。利用できる画面はレコード一覧画面のみ。
- `GET /k/v1/app/views.json`: `views.<一覧名>.type`(`LIST`/`CALENDAR`/`CUSTOM`)、`.id`、`.name` 等を返す。ビルトインの「すべて」ビューは含まれない(サンプルレスポンスにも該当エントリが存在しない)。
- `kintone.app.getFormFields()`(JS API): REST版 `GET /k/v1/app/form/fields.json` の `properties` と同等の値を返す。プラグイン設定画面・レコード一覧画面を含む主要画面で利用可能。
- `kintone.app.getQueryCondition()`: 現在の絞り込み条件のみ(`limit`/`offset`/`order by` を含まない)を取得。`kintone.app.getQuery()` はこれらを含む(今回は使わない)。
- `GET /k/v1/records.json`: 1回の取得上限は500件、`query` の `offset` 上限は1万件(このため `$id` 昇順ページングを採用)。
- `kintone.showLoading('VISIBLE' | 'HIDDEN')`: 全件取得中のローディング表示に使用。
- `kintone.showNotification('ERROR' | 'SUCCESS' | 'INFO', message)`: 全件取得失敗時のエラー通知に使用。
