# セルフルックアッププラグイン

## 機能

同一アプリ内の別レコードを検索し、一致した1件のフィールド値を自レコードへコピーする。kintone標準の
「ルックアップフィールド」は参照先を**別アプリ**にしか設定できないため、本プラグインは**同一アプリ内**での
ルックアップを可能にする。

- ルックアップ設定(以下「設定行」)を複数持てる。
- 設定行ごとに、キー一致条件(自レコードのキー値が他レコードのキー値に部分一致)・絞り込み条件(複数、
  AND結合)・フィールドマッピング(他レコード→自レコード、複数)を持つ。

## 技術アプローチ(REST必須)

同一アプリ内の「今表示していないレコード」を検索・取得するJavaScript APIは存在しないため、
`kintone.api()`(内部向けラッパー)経由で自アプリ自身に対し`GET /k/v1/records.json`を呼ぶ
(CLAUDE.md開発方針3: JS APIで実現できない場合のみRESTを使い、kintone自身への呼び出しに限り
`kintone.api()`を使用)。

## キー一致条件(確定・部分一致に変更)

`selfKeyFieldCode`(自レコードのフィールド)の値を含む(**部分一致・`like`演算子**)`otherKeyFieldCode`
(検索対象となる他レコードのフィールド、同一アプリなので通常は別のフィールドコードだが同じフィールド
コードを指定することも可)を持つレコードを検索する。これがクエリの主軸(kintone標準のルックアップ
フィールドにおける「検索キー」に相当)。

当初は完全一致(`=`演算子)のみだったが、user-test.mdフィードバック「LIKEで検索してなくない?部分一致で
ルックアップレコードを検索したい」を受けて部分一致に統一した(完全一致モードは廃止)。検索結果が複数件
ヒットした場合は、既存の「複数件一致した場合の採用ルール」節の通りモーダルで一覧表示しユーザーに選ばせる
ため、キー一致を1件に絞り込む必要がなくなった。

`otherKeyFieldCode`(検索先のキーフィールド)は、kintoneのクエリ言語で`like`演算子が使えるフィールド型
(**文字列(1行)・リンク**)のみ選択できる。ユニーク設定・レコード番号フィールドという制約は廃止した
(`like`演算子はレコード番号・数値等では使用できないため)。設定画面のプルダウンにもこの2種類の
フィールド型しか表示せず、`js/lib/config-validation.js`の保存前チェックでも`fieldInfoByCode`
(フィールドコード→`{type}`)を照合して二重に検証する。

## 絞り込み条件(9種類の演算子、確定)

元メモの「文字列完全一致、含む、等しい、以上、以下、より大きい、より小さい、同一月、同一日」を
以下の9種類の演算子として実装した(`js/lib/query-builder.js`)。

| 演算子 | 内部名 | kintoneクエリでの実装 | 備考 |
| :-- | :-- | :-- | :-- |
| 文字列完全一致 | `EXACT_MATCH` | `フィールド = "値"` | サーバー側で絞り込み |
| 含む | `CONTAINS` | `フィールド like "値"` | サーバー側で絞り込み |
| 等しい | `EQ` | `フィールド = "値"` | `EXACT_MATCH`と同じクエリになる(下記「判断記録」参照) |
| 以上 | `GTE` | `フィールド >= "値"` | サーバー側で絞り込み |
| 以下 | `LTE` | `フィールド <= "値"` | サーバー側で絞り込み |
| より大きい | `GT` | `フィールド > "値"` | サーバー側で絞り込み |
| より小さい | `LT` | `フィールド < "値"` | サーバー側で絞り込み |
| 同一月 | `SAME_MONTH` | (クエリでは表現不可) | 取得したレコード群に対し**クライアント側**で月が一致するかを判定する(`js/lib/client-filter.js`) |
| 同一日 | `SAME_DAY` | (クエリでは表現不可) | 取得したレコード群に対し**クライアント側**で日(day-of-month)が一致するかを判定する |

kintoneのクエリ言語には日付から月・日だけを取り出す関数がないため、`同一月`/`同一日`はサーバー側の
クエリに含められない。そのため、キー一致条件+サーバー側で表現可能な絞り込み条件でRESTから候補レコードを
取得したあと、`同一月`/`同一日`条件はJavaScript側で候補レコードをフィルタして判定する
(`判断記録.md`参照)。

## 絞り込み条件の値のソース(固定値/自レコードのフィールド参照、確定)

各絞り込み条件の比較値は、固定値または自レコードの別フィールドへの参照から選べる
(`records_to_subtable_import`と同じ設計)。

## 対応フィールド型のスコープ(確定・スコープ限定)

kintoneのクエリ言語は選択肢系フィールド(ドロップダウン・ラジオボタン・チェックボックス・複数選択)では
`=`/`>`等ではなく`in`/`not in`のみが使え、複数行文字列は`like`/`not like`/`is`/`is not`のみ使える等、
フィールド型ごとに使える演算子が異なる。本プラグインはキー一致条件・`EXACT_MATCH`/`EQ`/`GTE`/`LTE`/`GT`/
`LT`/`CONTAINS`の対象フィールドを**文字列(1行)・数値・日付・時刻・日時・リンク**に限定する(選択肢系
フィールドは対象外とし、設定画面のフィールド選択肢からも除外する)。

## 自己参照の除外(編集画面、確定)

編集画面(`edit`)では、検索クエリに`$id != 自レコードのレコード番号`を追加し、自分自身がヒットしないように
する。追加画面(`create`)ではレコード番号が未確定のため、この条件は付与しない。

## 複数件一致した場合の採用ルール(判断記録.mdの8・9番で変更)

`$id`昇順で並べ、条件を満たす候補をすべて取得する(`js/lib/client-filter.js`の`filterMatchedRecords`)。
件数にかかわらず(1件のときも)モーダルで一覧表示しユーザーに選択させ、選んだレコードだけを反映する
(自動での即時反映はしない、判断記録.mdの9番参照)。0件なら出力先を空文字列でクリアする(モーダルは
出さない)。`order by $id asc limit 500`で取得するため、501件目以降に条件を満たすレコードがあっても
検出できない点は変わらない(`判断記録.md`参照、`同一月`/`同一日`を含む設定でも同じ500件の枠内で
クライアント側フィルタする)。

## 発動タイミング・編集禁止の仕様(判断記録.mdの8・9番で変更)

- **ルックアップボタン(スペースフィールドに設置)を押したとき**にREST検索を実行する。元メモ通り
  「キー入力欄からフォーカスが外れたとき」の自動検索(`change`イベント)として実装したが、実機テストで
  `change`イベントがPromiseに対応しておらず`... is not allowed to return "Thenable" object.`エラーに
  なることが判明し、user-test.mdのフィードバックでボタン+モーダル方式への変更が指示されたため、
  発動方式をボタンクリックに変更した(判断記録.mdの8番に詳細)。
  - 検索結果が1件以上: 件数にかかわらず必ずモーダルに候補一覧を表示し、ユーザーが選択したレコードを
    反映する(判断記録.mdの9番。当初は1件時に自動反映していたが、「ボタンを押すとモーダルが出ると
    思っていたが即時反映されていた」というフィードバックを受けて、常にモーダルを経由する仕様に変更した)。
    モーダルに表示するフィールドは設定画面の「モーダルに表示するフィールド」(`modalFieldCodes`)で
    選べる(未設定時はフィールドマッピングの検索先フィールドにフォールバック)。
  - 検索結果が0件: 出力先フィールドをすべて空文字列でクリアし、アラートで通知する(選ぶ対象がないため
    モーダルは出さない)。
- 出力先フィールドは追加・編集画面(`create.show`/`edit.show`)で`disabled`にし、直接編集を禁止する
  (元メモ「挿入先の編集を禁止」)。
- キーフィールド(`selfKeyFieldCode`)はレコード一覧のインライン編集(`app.record.index.edit.show`)で
  `disabled`にし、一覧からの直接編集を禁止する(元メモ「元フィールドは一覧からの編集を禁止」)。
- レコード追加・編集画面では、`kintone.app.record.setFieldStyle()`で`selfKeyFieldCode`のフィールド名を
  赤字強調表示する(判断記録.mdの9番、user-test.mdフィードバック「赤くしてほしいのはレコード作成編集
  画面のキーフィールド」反映。設定画面上のラベルの赤字強調とは別)。

## 一致しなかった場合の挙動(確定)

条件に一致するレコードが1件もない場合、出力先フィールドをすべて空文字列でクリアする
(`wareki_date_format`等と同じ上書き方針)。

## クリアボタン(確定、ユーザーフィードバック反映)

本プラグインは値取得系のボタン(「検索して反映」)のみで、取得済みの値を取り消す手段が無かったため
「クリア」ボタンを追加した。専用の設定項目(スペースフィールド選択)は設けず、既存の
`buttonSpaceElementId`のスペースフィールドの中に「検索して反映」ボタンと並べて自動的に追加表示
される。主ボタンより一回り小さいスタイル(`slk-button-small`)にし、視覚的な優先度を下げている。

押すと`confirm()`で確認したうえで、自レコードの検索キー(`selfKeyFieldCode`)・転記項目の出力先
フィールドをすべて空文字列にクリアする(`js/desktop.js`/`js/mobile.js`の`clearAllFields`)。

### 既知の落とし穴: kintone.app.record.set()で値を空文字列にクリアすると、valueキー自体が消える

`biz_code_search`プラグインの開発時に実際の検証環境で確認した挙動: `kintone.app.record.set()`で
フィールドの`value`に空文字列を渡すと、その後の`kintone.app.record.get()`では`value`キー自体が
存在しなくなり(`undefined`)、`''`にはならない。本プラグインの`clearAllFields`はこの挙動を前提に
実装しており、クリア後の値を扱う既存コード(`setupLookupButton`のキー未入力チェック等)は元々
`!sourceField.value`のような偽値判定のみで、`.value`に対して`.trim()`等のメソッドを直接呼ぶ箇所が
無いため、影響を受けない。

## 設定画面(判断記録.mdの8・9番でスペースフィールド選択・モーダル表示フィールド選択を追加)

`kintone.plugin.app.setConfig()`にのみ保存する。設定画面の説明順は「①検索先を設定→②ルックアップ元
(自レコードのキー)を選ぶ→③絞り込み条件→④どこにルックアップするか(フィールドマッピング)→⑤モーダルに
表示するフィールド→⑥ボタンの設置場所」の流れに合わせている(user-test.mdフィードバック反映)。①②の
キーフィールドのラベルは赤字で強調する。

- 設定行を追加・削除できる。設定行ごとに:
  - 検索先のキーフィールド(`like`演算子が使えるフィールド型(文字列(1行)・リンク)のみ選択可)
  - 自レコードの検索キーとなるフィールド(対応フィールド型のみ選択可)
  - 絞り込み条件(複数追加・削除。フィールド・演算子・値ソース(固定値/自レコード参照)・値)
  - フィールドマッピング(他レコードのフィールド→自レコードのフィールド、複数)
  - モーダルに表示するフィールド(任意、複数追加・削除。`modalFieldCodes`。未設定時はフィールド
    マッピングの検索先フィールドにフォールバックする、判断記録.mdの9番)
  - ルックアップボタンを設置するスペースフィールド(`kintone.app.getFormLayout()`で取得したSPACER
    フィールドの`elementId`一覧から選択)
- 保存時に`js/lib/config-validation.js`でチェックする(検索先キーフィールドのlike対応型制約、
  ボタン設置スペースの必須・重複禁止、モーダル表示フィールドの空選択禁止を含む)。

## TDD

`src/js/lib/`配下の純粋ロジックはJestでユニットテストする(`pnpm test`)。

- `query-builder.js` — 設定行+自レコードの値から、サーバー側で表現可能な条件のkintoneクエリ文字列を組み立てる(値のエスケープ・自己参照除外`$id`条件を含む)
- `client-filter.js` — 取得した候補レコード群から、`同一月`/`同一日`等クライアント側限定の条件を満たす候補をすべて選ぶ(`filterMatchedRecords`)。先頭1件版の`pickMatchedRecord`はその薄いラッパー
- `field-mapping.js` — 一致レコード(またはnull)+フィールドマッピングから、自レコードへ書き込む値オブジェクトを組み立てる
- `config-store.js` — `kintone.plugin.app.getConfig()`/`setConfig()`のペイロードの読み書きとデフォルト値
- `config-validation.js` — 設定(設定行の配列)のバリデーション。検索先キーフィールドのlike対応型制約、ボタン設置スペースの必須・重複禁止を含む

kintone依存のグルーコード(`desktop.js`/`mobile.js`/`config.js`、`kintone.api()`呼び出しを含む)と、
DOM専用の`lookup-ui.js`(モーダル描画、jsdom未導入のためJestのnode環境ではテストできない)は
`src/e2e/*.e2e.test.js`(Puppeteer、`pnpm run test:e2e`)で実環境テストする(判断記録.mdの10番参照)。

- `config-screen.e2e.test.js` — 設定画面が開けること、検索先のキーフィールドがlike演算子の使える
  フィールド型(文字列(1行)・リンク)のみに絞り込まれること、ボタン設置スペースの選択肢が
  `kintone.app.getFormLayout()`から実際に出ることを確認する(公開サイト用スクリーンショットもここで
  撮影)。
- `record-lookup.e2e.test.js` — 実際にレコード追加画面でボタンを押し、モーダル経由で検索結果が

- `fixtures.js` — テスト用の検索先キーフィールド(文字列1行)・ボタン設置用スペース・検索対象の

## 実装

kintoneドキュメントMCPを参照しながら実装した。`app.record.edit.change.<フィールドコード>`イベントに
`recordId`(自己参照除外に使用)が含まれること、kintoneクエリ言語の演算子・エスケープ処理・
`order by`/`limit`仕様、`kintone.api()`の呼び出し方法を確認済み。セキュアコーディングガイドラインでの
リスクチェックは`security-checklist.md`を参照。
