# 判断記録

`plugin_idea_plan.md`(「## 5. 複数レコードサブテーブル化プラグイン」および「共通の前提・訂正事項」)に
明記がなく、実装時に新たに判断した事項を記録する。作業は止めずに進めた。

## 1. `@cybozu/eslint-config@25.0.1`のflat config APIの変更に対応

`fiscal_year_numbering/src/eslint.config.mjs`をコピー元として使う想定だったが、そのままコピーして
`pnpm run lint`を実行したところ、以下のエラーで起動不能だった。

```
TypeError: kintoneGlobalConfig is not iterable
```

原因調査の結果、`node_modules/@cybozu/eslint-config@25.0.1`では`flat/globals/kintone.js`や
`flat/lib/base.js`等の個別モジュールの公開形式が、fiscal_year_numberingのコード(`...kintoneGlobalConfig`と
配列としてspreadする書き方)が想定するものと変わっていた
(`flat/globals/kintone.js`はグローバル変数名のオブジェクト、`flat/lib/base.js`等は「呼び出すと設定配列を
返すファクトリー関数」になっていた)。この問題はfiscal_year_numbering自体の`pnpm run lint`でも
再現することを確認済み(同じ`package.json`バージョン指定・同じコピー元の設定ファイルのため、
既存プラグイン側も同時期に壊れていたとみられる。今回のタスクではfiscal_year_numbering側の修正は
スコープ外のため行っていない)。

**対応**: `records_to_subtable_import/src/eslint.config.mjs`は、パッケージが公式に提供している統合
プリセット`@cybozu/eslint-config/flat/presets/kintone-customize-prettier.js`
(内部で`[...kintone(), ...prettier()]`を返す。kintone向けグローバル+ベースルール+Prettier連携を
1つにまとめたもの)を使う形に書き換えた。また、`js/lib/*.js`(ブラウザ/Node両対応のUMD形式)と
`__tests__/*.test.js`(Jest)がkintone向けプリセットの標準グローバル(ブラウザ+kintone専用)には
含まれない`module`/`require`/`describe`/`test`/`expect`等を参照するため、追加の`languageOptions.globals`
設定ブロックを加えている。

※ほぼ同時期に別エージェントが`wareki_date_format`プラグインで同種の問題を発見し、同様の方針
(ファクトリー関数を呼び出す形への修正)で対応したとの共有を受けた。アプローチは独立に一致している。

## 2. 検索条件の演算子から`is`/`is not`(空欄判定)を除外

kintoneクエリの書き方(演算子と関数一覧)には、文字列(複数行)等向けの`is empty`/`is not empty`という
値を伴わない特殊な記法が存在する。本プラグインの`js/lib/query-builder.js`は「フィールドコード 演算子 値」の
3点セットを前提にした設計(`buildConditionClause(condition, resolvedValue)`)にしており、値を伴わない
演算子を混在させると値解決ロジック・UIの両方が複雑化するため、MVPでは対象外とした
(`=` `!=` `>` `<` `>=` `<=` `like` `not like` `in` `not in`のみサポート)。idea.mdに明記済み。

## 3. サブテーブル内フィールド・SUBTABLEフィールド自体を取得元/検索条件から除外

kintoneクエリの書き方では、サブテーブル内のフィールドを条件に含める場合`in`/`not in`のみが使え、
かつ通常のフィールドとは異なる特殊な指定になる。設定画面のフィールド選択UI・型互換性チェックの
複雑化を避けるため、取得元アプリの`SUBTABLE`型フィールド自体は選択肢から除外している
(`js/config.js`の`flattenSelectableFields()`)。

## 4. FILE(添付ファイル)フィールドは取得元・取り込み先のいずれも非対応

FILEフィールドの値はダウンロード専用の一時的な`fileKey`であり、単純な値コピーでは別レコードへの
再アップロードが必要になる(kintoneのフィールド形式ドキュメントに明記)。本プラグインのスコープ
(検索条件での絞り込み+サブテーブルへの行取り込み)を超えるため、`js/lib/type-compatibility.js`で
明示的に非対応とした。

## 5. サブテーブルへの新規行は`id`キーを付与しない

kintoneのフィールド形式ドキュメントには、REST APIでの登録/更新時「行の`id`を指定しない場合、
新規行として扱われる」という記載がある一方、JavaScript API(`kintone.app.record.set()`)については
同様の挙動が明記されていない(`get()`の戻り値の形式と同じ、とだけ記載)。本プラグインは実行のたびに
サブテーブルを全置換する仕様(既存行はすべて新規行として書き直す)であるため、`js/lib/row-mapper.js`は
安全側に倒して行に`id`キー自体を含めない(REST APIの挙動から類推した設計)。実環境での挙動確認は
CLAUDE.md項目6のPuppeteerテスト(スコープ外、別タスク)で行う。

## 6. ボタン設置には`getHeaderMenuSpaceElement()`/`getHeaderSpaceElement()`を使用

plugin_idea_plan.mdには「実行トリガー(ボタン設置画面)」とだけ書かれ、具体的な設置API・設置位置は
未指定だった。フォームにスペースフィールドを追加してもらう必要がないよう、
PCは`kintone.app.record.getHeaderMenuSpaceElement()`(レコード詳細/追加/編集画面で使える、
パンくずリストとメニューの間の領域)、モバイルは`kintone.mobile.app.getHeaderSpaceElement()`
(一覧/詳細/追加/編集画面で使える、ヘッダー下の領域)を使うことにした。どちらもイベントハンドラーを
`create.show`/`edit.show`のみに限定することで、対象画面以外にはボタンを設置しない。

## 7. 上限件数の打ち切り判定は「上限を超えた時点」まで1ページ分余分に取得する

`js/lib/limit-guard.js`の`hasReachedLimit(fetchedCountSoFar, maxRows)`は`fetchedCountSoFar > maxRows`
(超えたときのみtrue)としている。ちょうど上限件数に達しただけで打ち切ると、「本当にヒット件数が
上限を超えていたのか(続きがあったのか)」を区別できず、実行結果のtruncated(打ち切りが発生したか)
表示が不正確になるため。この設計により、上限ちょうどのヒット件数のときはtruncated: falseに、
上限を1件でも超えるときはtruncated: trueに正しく倒れる(ページサイズ500件単位で最大1ページ分、
上限を超えて多く取得することがあるが、実際にサブテーブルへ書き込む行は`applyLimit()`で
上限件数までに切り詰められるため、書き込み結果への影響はない)。

## 8. 型互換性チェックのポリシー(完全一致以外に許可する組み合わせ)

`js/lib/type-compatibility.js`は、完全一致以外に以下を「警告付きで許可」としている。

- 数値・日付・時刻・日時・単一選択・複数選択・ユーザー選択 → テキスト系列: 文字列化して格納
- 単一選択(ドロップダウン/ラジオボタン) → 複数選択(チェックボックス/複数選択): 1件の選択肢として格納

逆方向(複数選択→単一選択、テキスト→数値等)や、カテゴリーが全く異なる組み合わせ
(日付→日時、ユーザー選択→組織選択等)は不許可とした。表示形式の違いはあっても実用上意味のある
変換のみを許可し、値の意味が変わってしまう・情報が失われる変換は許可しない、という基準で判断した。

## 9. `.gitignore`はコピー元(fiscal_year_numbering)をそのまま使わず、`dist/`のみにした

`fiscal_year_numbering/src/.gitignore`をコピーしたところ、`package.json`・`pnpm-lock.yaml`・
`pnpm-workspace.yaml`・`eslint.config.mjs`まで無視対象に含まれていた。実際に確認したところ、
これらのファイルはfiscal_year_numbering側でも現在git管理下に入っていない(`git ls-files`で
ヒットしない)ことが分かった。CLAUDE.mdの構成説明では`<plugin_name>/src/`直下に`package.json`を
含む旨が明記されており、`private.ppk`と`src/dist/plugin.zip`のみがgit管理外とされている
(`private.ppk`はリポジトリルートの`.gitignore`の`*.ppk`パターンで、`node_modules/`もルートの
`.gitignore`で既にカバーされている)。このため、本プラグインの`.gitignore`は`dist/`のみを
記載する形に修正した(fiscal_year_numbering側の`.gitignore`修正はスコープ外のため行っていない)。
