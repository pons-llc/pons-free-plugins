# 複数レコードサブテーブル化プラグイン

## 機能

- 別アプリのレコードを検索条件で絞り込み、現在のレコードのサブテーブルへ行として取り込む
  (構想1「サブテーブル別アプリ挿入プラグイン」のちょうど逆方向のデータフロー)。
- 実行はレコード追加/編集画面に設置されるボタン起点。レコード表示時の自動取得は行わない
  (「ボタン起点が安全」というplugin_idea_plan.mdの判断を採用)。
- 検索条件は設定画面で複数指定でき、すべてAND条件で絞り込む。値は「固定値」または
  「現在のレコードのフィールドの値」のどちらかを選べる
  (例: 自レコードの`顧客コード`と一致する取得元レコードのみ取り込む)。
- サブテーブルへの書き込みは**全置換**(確定仕様): 実行のたびにサブテーブルをクリアし、
  取得結果で上書きする。既存行の一部だけ更新する、といった差分反映は行わない。
- 取り込み件数には上限があり(設定可能、既定300件)、ヒット件数が上限を超える場合は
  先頭から上限件数までを取り込み、超過した旨を実行時にアラート表示する。

## 技術アプローチ

- **取得元アプリのレコード取得**: 別アプリのレコード読み書きにJavaScript APIは存在しないため、
  `GET /k/v1/records.json`をREST APIとして`kintone.api()`(内部向けラッパー)経由で呼び出す
  (CLAUDE.md開発方針3、および共通の前提・訂正事項)。生の`fetch`/`XHR`は使用しない。
- **全件取得のページング**: `offset`・カーソルAPIは使わず、`$id`(レコードID)昇順ページング
  (`$id > 直前取得分の最大$id` + `order by $id asc limit 500`)を使う
  (plugin_idea_plan.mdの共通の前提・訂正事項に明記された確定方針)。
  実装は`js/lib/id-paging.js`(純粋関数)+`js/records-client.js`(kintone.api()を呼ぶグルーコード)。
  上限件数を超えた時点でページ取得を打ち切る(`js/lib/limit-guard.js`の`hasReachedLimit()`)。
- **サブテーブルへの書き込み**: `kintone.app.record.get()`/`set()`(モバイルは`kintone.mobile.app.record`)
  でサブテーブル(`SUBTABLE`)の`value`配列を丸ごと差し替える。JavaScript APIのみで完結し、
  対象アプリ自身へのREST書き込みは行わない。実際の保存(サーバーへの反映)はユーザーが
  画面で【保存】を押すまで行われない。
- **検索条件のクエリ合成**: `js/lib/query-builder.js`の純粋関数`buildQuery(conditions, currentRecord)`が、
  検索条件配列(フィールドコード・演算子・型・値ソース)と現在のレコードから、kintoneクエリ文字列
  (`order by`/`limit`は含まない)を組み立てる。値ソースが`RECORD_FIELD`の場合は
  `resolveConditionValue()`が現在のレコードから値を読み取る。構想1・4と同型の「値ソース切り替え」
  インターフェースを持つが、他プラグインのコードを直接参照せず独立実装している
  (plugin_idea_plan.mdの指示どおり、同種のインターフェースとして設計)。
- **フィールドマッピング**: `js/lib/row-mapper.js`が取得元レコード配列をサブテーブル行に変換する。
  テキスト系の列に数値・選択肢・日付などをマッピングする場合は文字列化して格納する
  (`stringifyForTextTarget()`)。新規行として追加するため、行の`id`は付与しない。
- **型互換性チェック**: `js/lib/type-compatibility.js`が取得元フィールド型とサブテーブル列の型の
  組み合わせを検証する。SUBTABLE/REFERENCE_TABLE/FILE等、値を単純コピーできない型は
  取得元・取り込み先のいずれにも指定できない。型が完全一致しない組み合わせでも、
  「テキストへの文字列化」「単一選択→複数選択」等、意味のある変換は警告付きで許可する。
  設定画面の保存時にも、互換性のないマッピングが残っている場合は保存をブロックする。

## 設定画面

`kintone.plugin.app.setConfig()`にのみ保存する。

- 取得元アプリID(REST APIでフィールド一覧を取得するボタン付き)
- 検索条件(取得元フィールド・演算子・値の種類[固定値/自レコードのフィールド参照]・値)を複数追加可能
- 書き込み先サブテーブル(このアプリの`SUBTABLE`フィールドから選択)
- フィールドマッピング(取得元フィールド → サブテーブル列、型互換性の警告/エラーを表示)
- 取り込み上限件数(既定300件)
- ボタンの表示文字列

設定画面自体のフィールド一覧取得は、このアプリ自身については`kintone.app.getFormFields()`
(JavaScript API)を使い、取得元アプリ(別アプリ)については`GET /k/v1/app/form/fields.json`を
`kintone.api()`経由で呼び出す(CLAUDE.md開発方針3: JavaScript API優先、別アプリはREST必須)。

## 画面ごとの挙動

- **レコード追加/編集画面**: `kintone.app.record.getHeaderMenuSpaceElement()`
  (モバイルは`kintone.mobile.app.getHeaderSpaceElement()`)にボタンを設置する。
  設定が未完了(取得元アプリID・書き込み先サブテーブル・フィールドマッピングのいずれかが空)の
  場合はボタン自体を設置しない。
- ボタン押下時: `kintone.showLoading('VISIBLE')`でローディング表示 →
  検索条件を現在のレコードの値で解決してクエリを合成 → `$id`ページングで取得元レコードを取得
  (上限件数+αで打ち切り) → フィールドマッピングで行に変換 → 上限件数で切り詰め →
  サブテーブルを全置換 → 結果(取り込み件数、打ち切りの有無)をアラート表示。
- レコード詳細画面・一覧画面ではボタンを設置しない(ボタン起点はレコード追加/編集画面のみ、との確定方針)。

## TDD

`src/js/lib/`配下の純粋ロジックはJestでユニットテストする(`pnpm test`)。

- `query-builder.js`: 検索条件からのクエリ文字列合成、値ソース(固定値/自レコード参照)の解決、
  エスケープ処理
- `id-paging.js`: `$id`昇順ページングのクエリ組み立て、次ページ判定、最大`$id`の算出
- `row-mapper.js`: 取得元レコード配列→サブテーブル行への変換、型ごとの値整形
- `limit-guard.js`: 取り込み件数の上限判定・打ち切りロジック
- `type-compatibility.js`: フィールド型の組み合わせの妥当性検証
- `config-store.js`: プラグイン設定の読み書き(`getConfig()`が`null`を返すケースを含む)

kintone依存のグルーコード(`records-client.js`, `subtable-writer.js`, `import-runner.js`,
`desktop.js`, `mobile.js`, `config.js`)は、CLAUDE.md項目6のPuppeteerによる実環境テストで
検証する(今回のスコープ外。別タスクで実施)。

## 未解決・利用時の注意点

- 検索条件の演算子は、kintoneクエリの書き方で値を伴わない`is`/`is not`(空欄判定)を
  MVPでは対象外にしている(値の入力欄が不要になる特殊な記法のため)。空欄判定が必要な場合は
  今後の拡張で対応する。
- フィールドマッピングの取得元にSUBTABLE内のフィールドは指定できない(サブテーブル内フィールドの
  クエリ条件指定には`in`/`not in`の特殊な書き方が必要なため、MVPでは対象外)。
- FILE(添付ファイル)フィールドは取得元・取り込み先のいずれにもサポートしない
  (値がダウンロード専用の`fileKey`であり、別レコードへの再アップロードなしに単純コピーできないため)。
- 取得元アプリのレコード閲覧権限がない場合、フィールド一覧の取得やレコード取得はエラーになる
  (実行時にアラートで通知するのみで、権限そのものはkintoneのアプリ設定に依存する)。
