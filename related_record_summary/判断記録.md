# 判断記録

`plugin_idea_plan.md`に明記されていない、実装中に必要になった判断を記録する。作業は止めずに進め、
判断内容と根拠をここに残す方針(タスク指示に基づく)。

## 1. 自アプリのフィールド設定取得はREST APIではなくJS APIを使う

`plugin_idea_plan.md`「4. 関連レコード集計プラグイン」の技術アプローチには「フォーム設定API(`GET /k/v1/app/form/fields.json`)
で対象フィールドの設定... を読み」とあるが、`mcp__claude_ai_kintone_doc__get_page`で
`/ja/kintone/docs/js-api/app/get-form-fields/`を確認したところ、`kintone.app.getFormFields()`(JS API)は
「フィールドを取得するREST APIの`properties`プロパティと同様の値」を返すと明記されており、`referenceTable`設定を
含めてJS APIだけで取得できることが確認できた。

CLAUDE.md開発方針3(「同じ目的を達成できる場合はJavaScript APIをREST APIより優先する」)に従い、
自アプリのフィールド設定取得は`kintone.app.getFormFields()`に統一した。REST APIは、別アプリ(参照先アプリ)の
レコード取得・自アプリの他レコードへの書き戻し・レコードカーソル操作など、JS APIが存在しない操作にのみ使っている。

設定画面(`config.js`)で参照先アプリの数値フィールド一覧(集計対象フィールドの候補)を取得する部分だけは、
他アプリのフィールド設定を読む必要がありJS APIが存在しないため、REST API(`GET /k/v1/app/form/fields.json`)を使っている。

## 2. `PUT /k/v1/records.json`のバッチ全体失敗仕様への対応(バッチ→個別フォールバック方式)

`plugin_idea_plan.md`は「書き戻し時にrevisionが競合したレコードはスキップして次へ進め、完了後にスキップ件数・
レコード番号一覧を結果表示する」とだけ確定しており、100件バッチでの書き戻し方法の詳細までは規定していない。

実装前に`mcp__claude_ai_kintone_doc__get_page`で`/ja/kintone/docs/rest-api/records/update-records/`を確認したところ、
「処理に失敗すると、リクエストに指定したレコードの更新はすべてキャンセルされます」と明記されていた。つまり
100件バッチのうち1件でもrevision競合すると、そのバッチ全体(最大100件)の更新が丸ごと失敗する。これでは
「競合したレコードだけスキップし、他は反映する」という確定事項を単純な100件一括PUTでは実現できない。

そこで、以下のフォールバック方式を実装した(`js/lib/batch-writer.js`)。

1. まず100件バッチで`PUT /k/v1/records.json`を試みる。
2. バッチが失敗(かつrevision競合と判定できるエラー)した場合のみ、そのバッチ内のレコードを1件ずつ
   `PUT /k/v1/record.json`で個別に送信し直す。
3. 個別送信で競合したレコードだけをスキップとして集計し、残りは正常に更新する。
4. 競合以外のエラー(権限エラー等)はバッチ・個別のどちらで発生してもそのまま再スローし、想定外の状態のまま
   処理を継続しない。

この方式により、通常時(競合なし)は100件バッチのAPI呼び出し回数のまま維持しつつ、競合が起きたバッチのみ
追加のAPI呼び出し(最大99回の個別PUT)が発生する。この追加コストは`js/lib/api-estimate.js`の見積もり式
(確定事項の計算式そのまま)には含めていない(見積もりは「競合が起きない前提」の概算という位置付け)。

## 3. revision競合の判定条件(ヒューリスティック、未検証)

`fiscal_year_numbering`の`isUniqueConstraintViolation`と同様の限界がある。kintone REST APIドキュメントに
revision競合時の正確な`code`値が網羅的に一覧化されていないため、既知の候補コード`GAIA_CO02`とメッセージ文言
(「リビジョン」/「revision」を含む)によるヒューリスティック判定にした。実環境(Puppeteer)での確認は
今回のタスクスコープ外のため未実施。security-checklist.mdの「個別確認事項」に記載済み。

## 4. 一括集計許可グループを複数指定可能にする(単一文字列ではなく配列)

`plugin_idea_plan.md`は「一括集計の実行許可グループ(複数)」と明記しているが、実装の参考にした
`fiscal_year_numbering`の`bulkNumberingGroupCode`は単一文字列だった。本プラグインは計画書の記載通り複数指定に
対応するため、`config-store.js`で`bulkGroupCodes`(配列)として保存し、設定画面ではカンマ区切りの1つのテキスト
入力欄で複数コードを入力する簡易UIにした(era-tableのような行追加UIにはしていない)。所属グループが
`bulkGroupCodes`のいずれか1つにでも一致すればボタンを表示する(OR条件)。

## 5. submit時集計の途中失敗時の再開(未決事項への暫定対応)

`plugin_idea_plan.md`の複雑性・リスク・未決事項に「途中失敗時の再開(何件目まで書き戻したか)の扱い」が
未決事項として残っていた。以下の暫定対応とした。

- **submit時集計**: 保存前の1回の集計・保存操作の中で完結するため、そもそも「途中」という状態が生まれない。
  集計に失敗したら保存自体をキャンセルする(`false`をreturn)ので、ユーザーは何も保存されていないことが明確
  であり、再度保存操作をやり直せばよい。
- **一括集計**: 進捗状態の永続化・再開機能は実装しない。処理中に例外でクラッシュした場合、それまでに書き戻し
  済みのレコードはそのまま残る(部分的に反映された状態になり得る)が、完了後の結果表示(成功件数・スキップ件数・
  スキップしたレコード番号一覧)はエラーで中断した場合には表示されない制約がある。ユーザーは一覧画面の
  絞り込み条件を再度指定して一括集計を再実行することで、未集計・未更新のレコードを再度対象にできる
  (収束的な設計: 集計処理自体は冪等であり、同じ入力なら同じ出力になるため、再実行しても不整合は生じない)。

## 6. 関連レコードのfilterCondに動的条件(LOGINUSER()等)が含まれる場合のクエリ再現互換性(未決事項への暫定対応)

`plugin_idea_plan.md`の複雑性・リスク・未決事項に「関連レコードの絞り込み条件(filterCond)には『ログインユーザー』
等の動的条件も書ける — クエリ再現の互換性をどこまで担保するか」が未決事項として残っていた。以下の暫定対応とした。

- `filterCond`の値は解釈・置換せず、文字列としてそのまま除外条件・一致条件とAND結合する(`js/lib/query-builder.js`)。
- `LOGINUSER()`のような関数呼び出しは、kintone REST API実行時にkintone側で評価される。このとき「誰の
  コンテキストで評価されるか」は「そのREST APIリクエストを実行したユーザー」になる。
  - submit時・詳細画面ボタン: レコードを保存・閲覧している本人が実行するため、直感的な挙動になる。
  - 一括集計: 一括集計を実行した人のコンテキストで評価される。対象レコードの本来の作成者・担当者とは
    異なる場合があり、`LOGINUSER()`を含む`filterCond`が設定されたフィールドを一括集計すると、
    レコードごとに想定と異なる集計結果になり得る。この点はsecurity-checklist.mdの「個別確認事項」に
    運用上の注意点として記載した(実行前の警告表示等の追加対応は今回のスコープでは実装していない)。

`wareki_date_format/src/eslint.config.mjs`で先に修正された内容(`kintoneConfig()`/`prettierConfig()`のように
