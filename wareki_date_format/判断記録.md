# 判断記録

`plugin_idea_plan.md`に明記がなく、実装時に暫定判断が必要だった事項を記録する。作業は止めずに進めた。

## 1. 書式プリセット・全角半角オプションを「ペアごと」に持たせるか、プラグイン全体で共有するか

- **問題**: タスク指示は「出力書式は『和暦のみ』『西暦和暦併記』の2種類をプリセットとして選べる」「全角にするかどうかを切り替えられるオプションを設ける」とあるが、これがプラグイン全体で1つの設定なのか、変換元→出力先のペアごとに個別設定できるのかが明記されていなかった。
- **選んだ暫定対応**: ペアごとに`preset`・`zenkaku`を持たせた(`js/lib/config-store.js`の`pairs`配列の各要素)。1つのアプリに複数の日付フィールドがあり、それぞれ用途が異なる(例: 画面表示用は「和暦のみ」、公文書用は「西暦和暦併記」)場合に対応できる方が実用性が高いと判断した。デメリットは設定画面がやや複雑になる程度で、実装コストの増分は小さい。
- **要ユーザー確認**: プラグイン全体で1つの書式に統一したいという要望であれば、設定画面・`config-store.js`のスキーマを変更する必要がある。

## 2. DATETIME系フィールドのタイムゾーン処理

- **問題**: kintoneのDATETIME/CREATED_TIME/UPDATED_TIME型の値はUTCのISO8601文字列(`"2012-01-11T11:30:00Z"`)で保存されており、画面にはログインユーザーのタイムゾーンで変換して表示される。一方、`Intl.DateTimeFormat`で和暦に変換する際は、どのタイムゾーンでの暦日(年月日)として扱うかを明示的に決める必要がある。kintoneのJavaScript APIには「ログインユーザーのタイムゾーンを取得する」ためのAPIが見当たらなかった(調査した範囲では)。
- **選んだ暫定対応**: `Asia/Tokyo`固定で暦日を計算する(`js/js/lib/wareki.js`の`DEFAULT_TIME_ZONE`。呼び出し側で`timeZone`オプションによる上書きも可能にしてある)。和暦を使う時点で日本国内の利用が前提になることが多く、kintoneの日本語ドメイン利用者の多くはタイムゾーン設定がAsia/Tokyoである可能性が高いという想定に基づく。
- **要ユーザー確認**: タイムゾーンがAsia/Tokyo以外のユーザーが実際に存在する場合、画面表示の日付とプラグインが計算する和暦の日付が異なる日にずれることがある(`security-checklist.md`の「タイムゾーンの扱い」に未解決事項として記録済み)。DATE型(タイムゾーンを持たない暦日そのもの)にはこの問題はない。

## 3. `app.record.create.show`/`edit.show`での先行計算を行わない

- **問題**: タスク指示は「`app.record.create.change.<field>`/`app.record.edit.change.<field>`イベントで即時反映、`create.submit`/`edit.submit`でも再計算する保険処理」とだけ書かれており、レコード表示直後(`show`イベント)の扱いは明記されていなかった。`fiscal_year_numbering`は`create.show`で先行計算するパターンを採用している。
- **選んだ暫定対応**: `show`イベントでの先行計算は行わないことにした。理由は、タスク指示に明記された2つのイベント(change/submit)だけで要件(値変更時の即時反映+保存時の保険)を満たせており、`show`時点での計算はスコープ外の追加機能と判断したため。変換元フィールドにデフォルト値が入っていて、ユーザーが一度もそのフィールドを操作せず保存した場合、画面上は出力先フィールドが一瞬空のまま保存されるが、`submit`イベントの保険処理で保存されるレコードの内容自体には反映される(`idea.md`に明記)。
- **要ユーザー確認**: 画面表示時点でユーザーに和暦の値を見せたい(保存前のプレビューが必要)という要望であれば、`show`イベントでの先行計算(`kintone.app.record.set()`使用)を追加する必要がある。

## 4. 出力先フィールドの重複設定を保存時エラーにする

- **問題**: 複数の変換ペアが同じ出力先フィールドを指定した場合の扱いがタスク指示に明記されていなかった。
- **選んだ暫定対応**: `js/lib/config-validation.js`の`validatePairs()`で、同じ出力先フィールドを複数ペアが指定している設定は保存時エラーとしてブロックする(「後勝ち」の挙動が実装依存になり予測しづらくなるため)。
- **要ユーザー確認**: 複数の変換元フィールドが状況に応じて同じ出力先フィールドを更新する、という用途を許可したい場合は、このバリデーションを緩めるか、優先順位を明示的に設定できるUIに変更する必要がある。

## 5. `@cybozu/eslint-config`のFlat Config APIがテンプレート(`fiscal_year_numbering`)と食い違っていた問題

- **問題**: `fiscal_year_numbering/src/eslint.config.mjs`をそのままコピーして`pnpm run lint`を実行したところ、`TypeError: kintoneGlobalConfig is not iterable`で失敗した。原因を調査したところ、`package.json`に固定されている`@cybozu/eslint-config@25.0.1`では、`flat/lib/base.js`・`flat/lib/kintone.js`・`flat/lib/prettier.js`・`flat/globals/kintone.js`が「設定配列そのもの」ではなく「呼び出すと配列を返すファクトリー関数」に変わっており(公式の`flat/presets/kintone-customize-prettier.js`が`[...kintone(), ...prettier()]`という呼び出し方をしているのを確認した)、`...kintoneGlobalConfig`のようにスプレッドでは動かなかった。`fiscal_year_numbering`側で実際に`pnpm run lint`を実行しても同じエラーで失敗することを確認済み(既存プラグイン側の`eslint.config.mjs`は現時点のロックされたバージョンでは動作しない状態になっている)。
- **選んだ暫定対応**: 本プラグイン(`wareki_date_format`)の`eslint.config.mjs`のみ、`kintoneConfig()`/`prettierConfig()`という関数呼び出し形式に修正した(`flat/globals/kintone.js`・`flat/lib/base.js`の個別importは、`kintone()`が内部で`base()`をkintone向けglobalsで呼び出し済みのため削除)。あわせて、Jest(`__tests__/**`)・Node CommonJS(`jest.config.js`, `eslint.config.mjs`)・isomorphic UMD風の`js/lib/**/*.js`向けに、`module`/`require`やJestのグローバル(`describe`/`test`/`expect`等)を追加で許可する設定ブロックを追加した(これらもkintone向けのブラウザグローバルだけでは`no-undef`エラーになったため)。
- **要ユーザー確認**: `fiscal_year_numbering/src/eslint.config.mjs`は本タスクのスコープ外のため修正していない。同プラグインで`pnpm run lint`を実行すると同じエラーになるはずなので、別タスクとして修正が必要。
