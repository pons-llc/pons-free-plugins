# プラグイン利用状況AI検索台帳プラグイン(plugin_catalog_builder)

`plugin_idea.md`の「プラグイン利用AI用バックデータ作成アプリプラグイン」(129行目)と
「プラグインAI検索作成プラグイン」(165行目、詳細版)を統合した仕様。社内のアプリ作成者が
「現在どのプラグインがどのアプリで使われているか」を横断的に把握し、目的に合ったプラグインを
AI検索(またはAI検索用データのダウンロード)で探せるようにする。

## 機能概要

1. **フィールド自動作成**: プラグイン設定画面で保存すると、このプラグインを追加したアプリ
   (以下「台帳アプリ」)に必要なフィールドを自動作成する。
2. **同期(取得してupsert)**: 台帳アプリの一覧画面に「プラグイン利用状況を同期」ボタンを表示し、
   押すとkintone環境全体の「インストール済みプラグイン」と「各プラグインを使っているアプリ」を
   REST APIで取得し、台帳アプリのレコードとしてUPSERTする(1プラグイン=1レコード、利用アプリは
   サブテーブルの行として持つ)。
3. **簡易AI検索(設定でON/OFF)**: ONの場合、台帳アプリのレコード一覧にAI検索UIを追加し、
   `site/`のAI検索(`js/ai-search.js`)と同じ方式(ブラウザ内完結のベクトル検索)でプラグインを検索できる。
4. **AI用データダウンロード(AI検索がOFFでも常時利用可)**: レコード全件をJSON形式の`.txt`として
   ダウンロードし、NotebookLMやChatGPTなど外部AIツールに読み込ませて検索させる用途に使う。

## 実データ・ドキュメントに基づく実装方針(kintoneドキュメントMCPで確認済み)

CLAUDE.md開発方針1に従い、以下はすべてkintoneドキュメントMCPで仕様を確認済み(未確認の推測実装はしない)。

### 使用するREST API(いずれもJavaScript APIでは代替不可、クロスアプリ・環境全体の情報のため)

| API | 用途 | 必要なアクセス権 |
| :-- | :-- | :-- |
| `GET /k/v1/plugins.json` | kintone環境にインストール済みのプラグイン一覧(`id`/`name`/`description`/`version`/`isMarketPlugin`) | なし |
| `GET /k/v1/plugin/apps.json?id=` | 指定プラグインを追加しているアプリ一覧(`id`/`name`) | **cybozu.com共通管理者** |
| `GET /k/v1/apps.json?ids[]=` | アプリの名称・説明・所属スペースID(`name`/`description`/`spaceId`) | 対象アプリのレコード閲覧または追加権限(権限がないアプリは結果に含まれず取得不可) |
| `PUT /k/v1/records.json`(`upsert: true` + `updateKey: plugin_id`) | 台帳アプリへの1プラグイン=1レコードのUPSERT(100件/リクエストで分割) | 台帳アプリのレコード追加・編集権限 |

### 重要な制約: 権限の実効的な境界は「cybozu.com共通管理者」

`GET /k/v1/plugin/apps.json`(プラグインを利用しているアプリの取得)は**cybozu.com共通管理者**権限を
必須とするAPIであり、この権限を持たないユーザーが同期ボタンを押してもAPIが403で失敗する。
ユーザー確認の結果、この上に重ねる「グループ(ロール)による実行者の絞り込み」は
**UI上の絞り込み(ボタンの表示/非表示)としてのみ実装**し、真の権限境界はkintone自身の
cybozu.com共通管理者権限に委ねる、という位置づけを明記する(`related_record_summary`の
`kintone.user.getGroups()`によるボタン表示制御と同じ設計・同じ限界。
`security-checklist.md`に「グループ制限はクライアント側の出し分けに過ぎない」旨を明記する)。

### フィールド自動作成: 実際に作成する対象は元メモのJSONの一部のみ(重要な訂正)

`plugin_idea.md`165行目の元メモには`レコード番号`/`作業者`/`作成者`/`更新者`/`ステータス`/`カテゴリー`/
`更新日時`/`作成日時`もフィールド定義JSONに含まれているが、これらはkintoneアプリに標準で存在する
システムフィールドであり、`POST /k/v1/preview/app/form/fields.json`(フィールドを追加する)の
ドキュメントには「次のフィールドは追加できません: ステータス、作業者、カテゴリー」と明記されている
(レコード番号・作成者・作成日時・更新者・更新日時は追加不要以前にAPIでの追加対象外= 既定で存在)。
よって**実際にAPIで作成するのは以下の5項目のみ**とする(元メモのJSONはkintoneアプリの
「フィールド設定エクスポート」的なダンプをそのまま転記したものと判断し、システムフィールドは除外する)。

| フィールドコード | type | 概要 |
| :-- | :-- | :-- |
| `plugin_id` | SINGLE_LINE_TEXT | プラグインID。`unique: true`(UPSERTの`updateKey`に使う)、`required: true`、`maxLength: 64` |
| `plugin_name` | SINGLE_LINE_TEXT | プラグイン名称 |
| `plugin_version` | SINGLE_LINE_TEXT | バージョン |
| `plugin_detail` | MULTI_LINE_TEXT | プラグインの説明(`plugins[].description`) |
| `pcb_apps_table`(表示ラベルは元メモ通り「テーブル」) | SUBTABLE | 利用アプリの一覧。内包フィールド: `app_name`(SINGLE_LINE_TEXT)/`app_id`(NUMBER)/`app_detail`(MULTI_LINE_TEXT、アプリの説明。所属スペース名が取れた場合は先頭に`[スペース名] `を付与) |

**フィールドコードの訂正(実環境E2Eテストで発見・確定)**: 元メモはこのSUBTABLEのフィールドコードも
`テーブル`としていたが、検証環境の`TEST_APP_ID_1`には元々`テーブル`という汎用的なコード名の
SUBTABLE(フィールドタイプ網羅用のサンプルフィールド)が既に存在しており、実際にE2Eテストを
実行したところ本プラグインが「既存構成を尊重」してこのフィールドを既存とみなし、
`app_name`/`app_id`/`app_detail`を含む本来のテーブルが永久に作成されない不具合が発生した。
`テーブル`のような汎用的な日本語ラベルをそのままフィールドコードに使うと、対象アプリに同名の
既存フィールドがあった場合に衝突するリスクがあるため、フィールドコードは`pcb_apps_table`という
一意な値に変更した(表示ラベルは元メモ通り「テーブル」のまま)。他の4項目(`plugin_id`等)は
英数字の一意な名前のため同様のリスクは無い。

### フィールド作成〜運用環境反映までの手順(確定)

`add-form-fields`は動作テスト環境(`/k/v1/preview/app/form/fields.json`)にしか反映されず、
運用環境に反映するには別途デプロイAPIの実行が必要(ドキュメント「補足」に明記)。設定画面の保存処理
(`config.js`の保存ボタンハンドラー)で以下を行う。

1. `GET /k/v1/preview/app/form/fields.json?app=` で現在のフィールド一覧+`revision`を取得する
   (動作テスト環境の設定取得にはアプリ管理権限が必要。設定画面自体がアプリ管理者にしか開けないため
   別途の権限チェックは不要)。
2. 上記5項目のうち、`code`が既存フィールドに存在しないものだけを抽出する(**冪等性**:
   2回目以降の設定保存では既存分を再作成しようとして400エラーにならないようにするため)。
   `pcb_apps_table`(SUBTABLE)については、テーブル自体は存在するが内包フィールドが不足している場合の
   追加は本プラグインのスコープ外とし、テーブル自体の有無のみで判定する(既存構成を尊重する)。
3. 追加対象が1件以上あれば`POST /k/v1/preview/app/form/fields.json`(`revision`指定)で
   動作テスト環境に追加する(この時点ではまだデプロイしない)。
4. `kintone.plugin.app.setConfig()`で設定を保存する。
5. `POST /k/v1/preview/app/deploy.json`(`revision`省略、検証をスキップ)でデプロイを実行し、
   `GET /k/v1/preview/app/deploy.json?apps[0]=`を`status`が`SUCCESS`になるまでポーリングする
   (`PROCESSING`の間は数百ms間隔で再試行、`FAIL`/`CANCEL`ならエラー表示してユーザーに再保存を促す)。

失敗時(フィールド作成・デプロイいずれか)は、エラーメッセージを表示する。

### 【実環境E2Eテストで発見・確定】プラグイン設定はフィールドと同じくデプロイが必要

**`kintone.plugin.app.setConfig()`で保存した値は、動作テスト環境(preview)にのみ反映され、
アプリを実際にデプロイするまでレコード一覧・詳細等の画面(`desktop.js`側の
`kintone.plugin.app.getConfig()`)からは古い値のまま見える。** これは`GET/PUT`
`/k/v1/preview/app/plugin/config.json`(検討中の新機能のREST APIだが、URLが`preview/`配下である
ことからも設定値がフィールドと同じpreview/production機構に乗っていることが分かる)という傍証に加え、
実際に検証環境で「許可グループを外して保存したのに、レコード一覧の同期ボタンが表示されたまま」という
不具合として顕在化し、`POST /k/v1/preview/app/deploy.json`を手動実行すると解消することを確認して
確定した。初期実装ではフィールド追加が無い場合(2回目以降の保存)にデプロイをスキップしていたため、
**フィールド追加の有無に関わらず、`setConfig()`のたびに必ずデプロイする**ように修正した
(フィールド追加分もこのデプロイでまとめて反映されるため、通常は1回のデプロイで済む)。

## 同期(取得してUPSERT)フロー

`related_record_summary`の一括集計ボタン(`js/bulk-summary.js`)と同じ設計パターンを踏襲する。

1. 一覧画面のヘッダー領域(`kintone.app.getHeaderMenuSpaceElement()`)に「プラグイン利用状況を同期」
   ボタンを表示する。表示条件は`kintone.user.getGroups()`で取得した所属グループが設定済みの
   許可グループに含まれる場合のみ(UI上の絞り込み、上記「権限の実効的な境界」参照)。
2. ボタン押下時、`kintone.showConfirmDialog()`でAPI実行回数の見積もり
   (「プラグイン数 + プラグイン数(利用アプリ取得) + 利用アプリ総数/100(アプリ情報取得) +
   対象レコード数/100(書き込み)」の概算式)を表示して確認を取る。
3. `kintone.showLoading('VISIBLE')` + `beforeunload`離脱防止を有効化する。
4. `GET /k/v1/plugins.json`を`offset`0始まり・`limit`100でページングして全プラグインを取得する。
5. 各プラグインについて`GET /k/v1/plugin/apps.json?id=`を実行し、利用アプリの`id`/`name`を取得する
   (403(cybozu.com共通管理者権限なし)の場合は、同期全体を中止しエラーメッセージを表示する)。
6. 全プラグインから収集したアプリIDの重複を除いた一覧に対し、`GET /k/v1/apps.json?ids[]=`を
   100件区切りで実行し、`name`/`description`/`spaceId`(スペース名は`spaceId`から別途
   `GET /k/v1/spaces/{id}.json`が必要になり複雑化するため、**初期スコープではスペース名の解決は行わず
   `spaceId`の有無のみ`app_detail`に反映しない**。将来的な拡張候補として`idea.md`に残す)を取得する。
   閲覧権限がなく結果に含まれなかったアプリは、`app_name`を空文字列・`app_detail`を
   `(アプリ情報を取得できませんでした)`とする。
7. プラグインごとに1レコード分の`record`オブジェクト(`plugin_name`/`plugin_version`/`plugin_detail`/
   `pcb_apps_table`の行配列)を組み立て、`PUT /k/v1/records.json`(`upsert: true`,
   `updateKey: { field: 'plugin_id', value: <プラグインID> }`)で100件ずつ書き込む。
   `pcb_apps_table`は行の`id`を指定せず全置換する(利用アプリ一覧は毎回の同期結果を正とする、
   ユーザーが手で編集する項目ではないため)。
8. 全プラグインの書き込みが終わったら結果(件数・失敗があれば内容)を`alert`で表示し、
   ローディング解除・離脱防止解除を行う。
9. **環境に存在しなくなったプラグインのレコードの扱い(確定)**: 削除・アンインストールされた
   プラグインの既存レコードは自動削除しない(UPSERTは追加/更新のみで削除は行われない)。
   台帳アプリ側で棚卸しのために手動確認・削除する運用とする(自動削除は「使われていないと
   誤検知して消してしまう」リスクがあるため意図的にスコープ外)。

## 簡易AI検索(設定でON/OFF、CDN例外)

### 例外事項(強調・必読)

**本プラグインに限り、CLAUDE.md開発方針9(外部パッケージ・外部通信を一切使わない)の例外として、
CDN経由での外部ライブラリの動的読み込みを認める**(ユーザー承認済み、元アイデア通り)。
`excel_report_export`の「外部ライブラリはビルド時にバンドルし、CDN読み込みはしない」という前例とは
**異なる判断**であることに注意(あちらは外部通信そのものを禁止、本プラグインは簡易AI検索ON時に限り
実行時の外部通信を許容する)。

- 使用ライブラリ: `site/js/ai-search.js`と同じ`@huggingface/transformers`
  (`https://cdn.jsdelivr.net/npm/@huggingface/transformers@<バージョン>/+esm`、
  モデルは`Xenova/multilingual-e5-small`)を`import()`で動的読み込みする。
- **注意事項としてsecurity-checklist.mdに強く明記する内容**:
  - 簡易AI検索をONにすると、レコード一覧画面を開いたユーザーのブラウザから`cdn.jsdelivr.net`へ
    通信が発生し、CDN側で配信内容が改ざん・置き換えされた場合の影響を受け得る(サプライチェーンリスク)。
  - バージョンは`@4.2.0`のように**完全固定**し、`@latest`のような可変指定をしない。
  - 台帳アプリのレコード内容(プラグイン名・説明・利用アプリ名など)がブラウザのメモリ上でCDN配信
    コードの処理対象になる(機密情報を含むフィールドをこのアプリに追加しないよう運用上注意する)。
  - CDNが利用不可・低速な環境では検索機能自体が使えない(ダウンロード機能はCDN非依存で常時使える
    ため、そちらを代替手段として案内する)。
- 実装は`site/js/ai-search.js`のコーパス生成部分を、台帳アプリの全レコード取得
  (`$id`昇順ページングの全件取得、共通前提を踏襲)に置き換える形で作る
  (静的コーパスの代わりに`GET /k/v1/records.json`の結果を使う点のみ異なる)。

### 検索実行時のフリーズ対策(ユーザー報告により確定)

初回リリース版は、初回検索時に台帳アプリの全レコードをまとめて1回の`extractor()`呼び出しで
埋め込みベクトル化しており、この単一のWASM計算がメインスレッドを長時間ブロックし、実際に
「検索窓に値を入れて検索するとブラウザが固まる」という不具合として利用者から報告された。
`js/desktop.js`の`getCorpusEmbeddings`を8件ずつのバッチ処理に変更し、バッチの間で
`setTimeout(0)`によりイベントループへ制御を一度返すことで、1回あたりのブロック時間を短く抑えた
(合計の計算時間自体は変わらないが、ダイアログの応答性を保ち「固まった」ように見える状態を避ける)。
あわせて、モデルダウンロード中は`progress_callback`から進捗(%)を取得してプログレスバーに表示し、
初回検索の埋め込み計算中もバッチごとの進捗(%)を表示するようにした(site/js/ai-search.jsの
プログレスバー表示と同じ考え方)。

## ダウンロード機能(AI検索OFFでも常時利用可)

- レコード一覧画面に「AI用データをダウンロード」ボタンを表示する(AI検索設定のON/OFFに関わらず表示)。
- `$id`昇順ページングで全レコードを取得し、`plugin_id`/`plugin_name`/`plugin_version`/`plugin_detail`/
  利用アプリ一覧(サブテーブル)をJSON配列に変換し、Blob + `a[download]`で`.txt`拡張子として
  ダウンロードする(NotebookLM・ChatGPT等へのアップロードを想定した拡張子選択。中身はJSON文字列)。
- 外部通信は発生しない(CDN例外はAI検索機能のみに閉じる)。

## 設定画面

- AI検索機能のON/OFF(チェックボックス)
- 同期ボタンを表示するグループ(複数選択、`kintone.plugin.app.setConfig()`保存時にコード配列で保持)
- (フィールド自動作成は設定項目ではなく、上記の通り保存時に自動実行する固定動作)

## エッジケース・未決事項

- 同一kintone環境に複数の台帳アプリ(本プラグインを追加したアプリ)が存在する場合の重複は想定しない
  (1環境1台帳アプリの運用を前提とする。複数ある場合はそれぞれが独立してプラグイン一覧を丸ごと
  取得するため、内容は基本的に同一になる)。
- プラグインストアのプラグイン(`isMarketPlugin: true`)と自作プラグインの区別は`plugin_detail`等には
  出さず、将来の拡張候補とする(初期スコープでは全プラグインを区別なく扱う)。
- 同期の実行中にブラウザを閉じた場合、途中まで書き込まれたレコードが残る(削除は行わないため
  「一部だけ更新された」状態になり得る)。再実行すれば冪等に上書きされるため、致命的にはならない
  という前提で許容する。

## TDD方針

`src/js/lib/`配下の純粋ロジックをJestでテストする。

- `field-diff.js` — 現在のフィールド一覧(`GET .../form/fields.json`の`properties`)と、作成すべき
  5項目の定義から、追加が必要なフィールドのみを抽出する(冪等性の中核ロジック)
- `plugin-record-mapper.js` — `plugins.json`・`plugin/apps.json`・`apps.json`の3種のレスポンスから、
  UPSERT用の`records`配列を組み立てる(利用アプリが1件も無いプラグイン、アプリ情報が取得できない
  ケースを含む)
- `api-estimate.js` — 同期実行前の確認ダイアログに出すAPI実行回数見積もりの計算式
- `deploy-poller.js` — デプロイステータスのポーリング判定ロジック(`PROCESSING`は継続、
  `SUCCESS`/`FAIL`/`CANCEL`で終了、`FAIL`/`CANCEL`はエラー扱い)を、注入した`getStatus`関数に対して
  確定的にテストする(`resolve-org-info.js`と同様の「取得関数を注入してオーケストレーションだけを
  ユニットテストする」設計)

kintone依存のグルーコード(`config.js`のフィールド作成・デプロイ呼び出し、`desktop.js`の同期ボタン・
AI検索・ダウンロード)は`src/e2e/*.e2e.test.js`(Puppeteer)で実環境テストする。
