# プラグイン構想 実装前計画書

[plugin_idea.md](plugin_idea.md) の6つの構想メモを、実装着手前の計画として整理したもの。
各プラグインの正式な `idea.md` / `security-checklist.md` を書く際の出発点として使う。
API名・仕様はすべて kintoneドキュメントMCP(`kintone_doc`)で確認済み(2026-07-09時点)。

## 共通の前提・訂正事項

構想メモに書かれたAPI名は一部実在の名前と異なるため、以下に訂正する(実装時はこちらを使う)。

| メモの表記 | 実際のAPI | 備考 |
| :-- | :-- | :-- |
| `kintone.app.createDialog()` | `kintone.createDialog(config)` | PC画面専用。Promiseで`{show, close}`を返し、`beforeClose`で閉じる前の割り込みが可能。`config.body`のElementはそのまま挿入されるためサニタイズ必須。モバイルはボトムシート作成APIが別途ある |
| `kintone.app.showConfirmDialog()` | `kintone.showConfirmDialog(config)` | PC画面専用。Promiseが`OK`/`CANCEL`/`CLOSE`で解決。モバイルは`kintone.showConfirmBottomSheet` |
| `kintone.app.showLoadingModal()` | `kintone.showLoading('VISIBLE' / 'HIDDEN')` | 表示中はユーザー操作をブロック。モバイルは`kintone.mobile.showLoading` |

共通の技術的前提:

- **別アプリのレコード読み書きにJS APIは存在しない**。クロスアプリのレコード取得・登録・更新はすべて`kintone.api()`経由のREST API(`/k/v1/records.json`等)になる(構想1・3・4・5・6すべてが該当)。
- REST APIの主な上限: レコード取得は1リクエスト500件、登録・更新は1リクエスト100件、`/k/v1/bulkRequest.json`は1回20操作まで。
- **全件取得方針(確定)**: 原則`offset`・カーソルAPI(`/k/v1/records/cursor.json`)は使わない。`$id`(レコード番号)昇順で`order by $id asc`+`query`条件、1回目は素の条件、2回目以降は`$id > 直前取得分の最大$id`を条件に追加して500件ずつページングする(offset上限1万や生成コストの問題を回避できるkintone推奨パターン)。全プラグイン共通のページング関数として`src/js/lib/`に切り出し、コピーで使い回す。
  - **例外(確定・ユーザー指示)**: 構想4(関連レコード集計)の「一覧画面からの一括集計」のように、**対象レコード列挙後に自レコードへ書き戻す**フロー(他ユーザーの同時編集で対象集合が動く/`$id`ページング中に増減がずれるリスクがある)では、代わりに**レコードカーソルAPI**(`POST /k/v1/records/cursor.json`→`GET /k/v1/records/cursor.json`を`next:false`まで繰り返し)を使う。カーソルは作成時点の対象レコード集合を固定するため、`$id`ページング中の追加・条件外れによる二重集計・取りこぼしを避けられる(ただしフィールド値自体は取得時点の最新値が返る点に注意)。
    - **カーソルAPI利用時の注意点(ドキュメント記載必須)**: ①同時に作成できるカーソルは1ドメイン10個まで、超過時は他のカーソルが完了/削除/失効するまで新規作成不可。②カーソルの有効期限は最終アクセスから10分、作成自体も5分でタイムアウト。③`next:true`でも次のレスポンスの`records`が空になることがある(ループはnextで判定し、空配列でも継続)。④キーワード検索(`like`/`not like`)を条件に含む場合、10万件時点で打ち切られ`X-Cybozu-Warning`ヘッダーが付く。⑤全件取得完了でカーソルは自動削除されるが、途中中断時は`DELETE /k/v1/records/cursor.json`で明示的に削除する。⑥書き戻し(`PUT`)時に対象レコードのrevisionが競合した場合は、そのレコードをスキップして次へ進め、完了後にスキップ件数・レコード番号一覧を結果表示する。
- `PUT /k/v1/records.json`は`upsert: true` + `updateKey`(重複禁止設定済みの文字列1行または数値フィールドのみ指定可)でUPSERTが可能(構想1の「キー指定更新」はこれで実現できる)。
- プラグイン設定(`kintone.plugin.app.setConfig`)は1値65,535文字・合計256KBまで(構想6のテンプレート保存方式に直接影響)。
- 関連レコード一覧フィールド(REFERENCE_TABLE)は**値の取得・登録・更新が一切できない**。フィールド設定(参照先アプリ・絞り込み条件)をフォーム設定APIで読み、参照先アプリへ自分でクエリする必要がある(構想4の中核制約)。

## 1. サブテーブル別アプリ挿入プラグイン

### 概要(整理した要件)

- 元アプリのサブテーブルを、別アプリへ「1行=1レコード」として登録する。
- 転送対象はサブテーブル内の列に加え、元レコードのテーブル外フィールドも含められる(全行共通の値としてコピー)。
- サブテーブル内の特定列をキーに指定すると、挿入先アプリの対応キーを基準に更新(UPSERT)できる。
- 実行タイミングは2種類:
  - **保存(submit)時**: レコードの値に対するAND/OR複合条件で発動可否を判定。転送の実施有無と転送結果を表示する。
  - **任意タイミング**: `kintone.createDialog()`によるモーダルで「一括転送/選択転送/キャンセル」を選べる。
- 転送成功時に、元レコードの特定フィールドへ特定の値を書き込める(転送済みフラグ等の用途)。

### 技術アプローチ

- **転送先への書き込み**: REST必須。新規登録は`POST /k/v1/records.json`、キー更新は`PUT /k/v1/records.json`の`upsert: true` + `updateKey`。いずれも100件/リクエストなので行数に応じて分割。
- **submit時**: `app.record.create.submit` / `app.record.edit.submit`イベント(Promise対応)内で条件判定→REST実行。失敗時に`event.error`で保存自体を止めるかは設計判断(下記)。
- **任意タイミング**: 詳細画面(`app.record.detail.show`)等にボタンを設置し、`kintone.createDialog()`で行選択UI(チェックボックス付きの行一覧)を自前描画。
- **条件判定エンジン(AND/OR)**: kintoneに依存しない純粋ロジックとして`src/js/lib/`に切り出し、Jestでテスト(構想4・5でも再利用コピー可能な形にする)。
- 成功時のフィールド書き込みは、submit時なら同イベント内で`event.record`を書き換え(同一保存に含まれる)、任意タイミングなら追加の`PUT`(自レコード更新)。

### 設定画面スケッチ

- 対象サブテーブル、転送先アプリID
- フィールドマッピング(サブテーブル列→転送先フィールド、テーブル外フィールド→転送先フィールド)
- 更新キー(サブテーブル列 と 転送先の重複禁止フィールド のペア)、UPSERTを使うか否か
- 発動条件ビルダー(フィールド・演算子・値の組をAND/ORで複数)
- 実行タイミング(submit / 手動ボタン / 両方)
- 成功時アクション(書き込み先フィールド + 値)

### 複雑性・リスク・未決事項

- **転送失敗時に保存を止めるか**: submitイベント内でREST失敗→`event.error`で保存中止にするか、保存は通して結果だけ表示するか。中止にしない場合「保存済みなのに転送されていない」状態が生まれる。
- **重複転送の防止**: 同じレコードを2回保存したら行が二重登録される。更新キー運用を必須にするか、行ごとの転送済み管理(どこに持つ?)をするか。
- **削除された行の扱い**: 元テーブルから行を消したとき、転送先レコードを消すのかは触れられていない(スコープ外とするのが無難だが要確認)。
- 転送先アプリへのアクセス権(追加・編集権限)がないユーザーが保存した場合の挙動・エラー表示。
- `kintone.createDialog()`はPC専用のため、モバイルは非対応とするかボトムシートで対応するか。
- 部分失敗時のロールバック: 100件ずつの分割実行中に途中で失敗した場合、先行分は登録済みになる。

### 規模感: **大**

条件エンジン + マッピングUI + ダイアログUI + UPSERT + 失敗時挙動と、独立した難所が多い。

## 2. 西暦和暦変換プラグイン

### 概要(整理した要件)

- 日付/日時フィールドを選択し、その値を和暦(または西暦・和暦併記)の文字列として指定した文字列フィールドへ自動入力する。
- 変換ロジックは既存の暦の対応を維持しつつ、**将来の元号変更に自動対応**できる構造にする。

### 技術アプローチ

- 変換は`Intl.DateTimeFormat('ja-JP-u-ca-japanese', { era: ... })`で行う。元号テーブルをコードに埋め込まないため、新元号はブラウザ(ICU)の更新で自動追従する — 「未来の元号変更に対応」の最も堅い実現方法。外部ライブラリ不要。
- 入力タイミング: `app.record.create.change.<フィールドコード>` / `app.record.edit.change.<フィールドコード>`で日付変更時に即時反映し、保険として`create.submit` / `edit.submit`でも再計算。
- 変換ロジックは`src/js/lib/`の純粋関数(Date→文字列)としてTDD。`fiscal_year_numbering`と同型の構成で実装できる。
- REST API不要。JS APIのみで完結する見込み。

### 設定画面スケッチ

- 変換元フィールド(日付/日時)と出力先フィールド(文字列1行)のペアを複数設定
- 出力書式(例:「令和7年7月9日」「R7.7.9」「2026年(令和7年)7月9日」等のプリセット or パターン)

### 複雑性・リスク・未決事項

- メモの「西暦和暦を入力」が「和暦のみ」か「西暦+和暦の併記」かが曖昧。書式プリセットの範囲を先に決めたい。
- 元号の境界日(改元当日)の扱い、明治以前の日付の扱い(Intlは対応するが表示要件として必要か)。
- change/submitイベント経由でのみ動くため、**RESTや一覧のインライン編集で日付が変わった場合は追従しない**。ここを許容するか(一覧一括変換機能を足すか)は要判断 — 足すなら規模が一段上がる。
- 出力先フィールドをユーザーが手で書き換えられてしまう問題(上書き方針)。

### 規模感: **小**

純粋ロジック中心・REST不要・画面要素なし。TDDの練習台として最適。

## 3. ガントチャートプラグイン

### 【2026-07-09 方針変更】カスタマイズビュー→通常の一覧(表形式)に変更

当初案の「カスタマイズビュー」ではなく、**通常の一覧(表形式, `viewType: 'list'`)のヘッダー領域**にガントチャートを描画する方式に変更。ユーザー指示による判断。

- 描画位置: `kintone.app.getHeaderMenuSpaceElement()`(PC専用、レコード一覧画面全般で利用可、集計アイコンの右側の要素を取得)を使う。カスタマイズビュー専用の`html`差し込みではなく、通常の一覧を含むどの一覧タイプでも同じ仕組みで描画できる。
- `app.record.index.show`イベントの`viewType`が`'list'`(表形式)のときに対象と判定し、`event.viewId`で設定を引き当てる方式は維持。
- **注意(kintone REST API制限、ユーザー指摘・要対応)**: `GET /k/v1/app/views.json`のレスポンスには、ビルトインの「すべて」ビュー(view ID = `20`など、ユーザーが作成していないデフォルト一覧)が**含まれない**(ドキュメントのサンプルレスポンスにも「すべて」相当のエントリがなく、ユーザー作成分のみが列挙される)。そのため設定画面で「対象一覧」をAPI取得結果からのみ選ばせると「すべて」ビューを選択できない。対応方針: 設定画面では「すべて(デフォルト)」を固定の選択肢として別枠で用意し、`event.viewId`がAPI取得済み一覧のどれにも一致しない場合はこの「すべて」設定にフォールバックする。
- カスタマイズビューごとに表示設定を切り替えられる → 一覧(list)ごとに表示設定を切り替えられる、に読み替え。
- デフォルトの並び順は一覧の表示順に従う。作業者/組織/ユーザー/ドロップダウン/ラジオボタンの各フィールドでグループ分けでき、既定のグループ分けと利用可能なグループ分けを設定できる。
- バー内に表示する項目、色分けに使う項目を設定できる。
- 終了日未設定の行は1日幅で表示、開始日未設定の行は最下部の「未仕訳」グループに表示する。

### 技術アプローチ

- **描画**: 外部ライブラリ不可のため、SVGまたはDOM+CSS Gridでガントを自前実装する(本プラグイン最大の作業量)。時間軸のスケール(日/週/月)・スクロールも自前。ヘッダー領域(`kintone.app.getHeaderMenuSpaceElement()`)に描画するため、一覧本体の表(行)には手を入れず独立したガント表示ブロックとして挿入する。
- **イベント**: `app.record.index.show`。`event.viewType === 'list'`(表形式)で対象一覧か判定し、`event.viewId`で設定を引き当てる。表形式は`event.records`が常に「現在ページ分」のレコード配列(空にはならない)。
- **全件取得**: `kintone.app.getQuery()`/`kintone.app.getQueryCondition()`で現在の絞り込み(クエリ)を取得し、`$id asc`+`$id > 直前最大$id`のページング(共通前提の全件取得方針)で`GET /k/v1/records.json`を500件ずつ呼ぶ。ここだけREST必須。offset・カーソルAPIは使わない。
- **ビューの列挙**: 設定画面では`GET /k/v1/app/views.json`で`type: LIST`のビュー一覧を取得。ただしビルトインの「すべて」ビューはレスポンスに含まれないため、「すべて(デフォルト)」を固定選択肢として別途用意する(上記【方針変更】参照)。
- グループ分け・色分け対象のフィールド列挙は`kintone.app.getFormFields()`系で型を絞って取得。

### 設定画面スケッチ

- 一覧ごとのタブまたはリスト(対象の表形式一覧の選択。「すべて(デフォルト)」を含む)
- 開始日・終了日フィールド(日付/日時)
- グループ分け: 利用を許可するフィールドの複数選択 + 既定のグループ分け
- バー内表示項目(複数フィールド)、色分けフィールドと色の割り当て
- 全件取得ボタンの有効/無効

### 複雑性・リスク・未決事項

- **対象一覧自体を誰が作るか**: 管理者が手動で一覧を作成する前提(確定・ユーザー回答済み)。プラグインによる自動作成は行わない。
- **描画性能と件数上限**: 全件取得時に数千件を描画すると重い。仮想スクロールを入れるか、取得上限を設けるか。
- バーのドラッグによる期間編集はメモにない → **表示専用**と割り切る想定でよいか要確認(編集対応だと規模が跳ね上がる)。
- 一覧が削除・変更されたときの設定の後始末(viewIdが浮く)。
- 一覧の表示順(order by)とグループ分けの併用時の並び順ルール。
- モバイル(`mobile.app.record.index.show`)対応の要否。`kintone.app.getHeaderMenuSpaceElement()`はPC専用なので、モバイル対応する場合は別途モバイル用の要素取得APIを調査する必要がある。

### 規模感: **大**(6案中最大級のUI実装)

ガント描画そのものをvanilla JSで作る工数が支配的。設定画面もビュー単位で最多クラス。

## 4. 関連レコード集計プラグイン

### 概要(整理した要件)

- 関連レコード一覧フィールドを選択し、その参照先レコードの「件数・合計・平均」を自レコードの指定フィールドへ書き込む。
- 関連レコードフィールドごとに集計種別と書き込み先を設定でき、除外条件も指定できる。
- 発動条件は複数設定可能: (a) submit時、(b) 詳細画面での任意実行、(c) 一覧画面からの一括集計。
- 一括集計は現在のクエリ条件のレコードを対象とし、実行可能者をグループで制限する。
- 実行前にAPI実行回数の見積もり(計算方法を明示)を`kintone.showConfirmDialog()`で確認し、実行中は`kintone.showLoading()`+ページ離脱防止を行う。

### 技術アプローチ

- **中核制約**: 関連レコード一覧フィールドの値はAPIで取得不可(ドキュメント明記)。よって、フォーム設定API(`GET /k/v1/app/form/fields.json`)で対象フィールドの設定(参照先アプリ・関連付けフィールド・絞り込み条件)を読み、**参照先アプリへ自分で`GET /k/v1/records.json`のクエリを組み立てて**集計する。REST必須。
- 除外条件は、フィールド設定の絞り込み条件(filterCond)とAND結合したクエリ文字列に合成する(クエリ組み立ては純粋ロジックとして`src/js/lib/`でTDD)。
- 件数だけなら`totalCount: true`で取得コストを抑えられる。合計・平均は対象フィールドを`fields`指定で取得して集計。
- **一括集計(確定・方式変更)**: 対象レコードの列挙は`kintone.app.getQuery()`で現在の絞り込み条件を取得し、それを`query`にしてレコードカーソルAPI(`POST`→`GET`ループ、共通前提の例外規定を参照)で列挙する(`$id`ページングではなくカーソルAPIを使う)。レコードごとに参照先を集計→`PUT /k/v1/records.json`で100件ずつ書き戻し。書き戻し時にrevision競合(409)が起きたレコードはスキップして次へ進み、完了後にスキップ件数・レコード番号一覧を結果表示する。
- **グループ制限**: `kintone.user.getGroups()`(JS API、キャッシュあり・50回/分の制限)でログインユーザーの所属グループを判定し、一括集計ボタンの表示可否を切り替える。
- **API回数見積**: 「対象レコード数 × 関連フィールド数 ÷ 500件 + 書き戻し回数」といった式を確認ダイアログの本文に明示する(`config.body`は文字列のみなので表現はテキストベース)。
- 離脱防止は`beforeunload`(標準ブラウザAPI)。`kintone.showLoading()`は表示中ユーザー操作をブロックする仕様なので併用で十分。

### 設定画面スケッチ

- 集計設定の行(複数): 関連レコードフィールド / 集計種別(件数・合計・平均) / 集計対象フィールド(合計・平均時) / 書き込み先フィールド(数値) / 除外条件
- 発動条件: submit時 ON/OFF、詳細画面ボタン ON/OFF、一覧一括 ON/OFF
- 一括集計の実行許可グループ(複数)

### 複雑性・リスク・未決事項

- **グループ制限はクライアント側の出し分けに過ぎない**(本当の権限制御ではない)。悪意ある操作を防ぐものではないことをドキュメントに明記するか、割り切りの確認が必要。
- **参照先アプリの閲覧権限がないユーザーが実行した場合(確定)**: エラーとして書き込みを中止する(集計結果が実態とズレたまま書き込まれることを防ぐ)。事前に`kintone.api()`のレスポンスや権限起因で件数が想定と乖離していないかを検証する手段は持たないため、実行前の注意書き表示で運用上の担保とする。
- **一括集計中のrevision競合(確定)**: 上記「一括集計」の技術アプローチのとおり、列挙はカーソルAPIで安定化し、書き戻し時の個別競合はスキップ+結果一覧表示で対応する。
- submit時集計は保存前に参照先アプリへクエリするため保存が遅くなる。関連フィールド数が多いアプリでの体感劣化。
- 関連レコードの絞り込み条件(filterCond)には「ログインユーザー」等の動的条件も書ける — クエリ再現の互換性をどこまで担保するか。
- 途中失敗時の再開(何件目まで書き戻したか)の扱い。

### 規模感: **中〜大**

集計・クエリ合成ロジックはTDD向きで手堅いが、一括実行系(見積・進捗・失敗処理)の作り込みが重い。

## 5. 複数レコードサブテーブル化プラグイン

### 概要(整理した要件)

- 別アプリのレコードを検索条件で絞り込み、現在のレコードのサブテーブルへ行として取り込む。
- 検索条件は設定で指定できる。

(6案の中で最もメモが短い。構想1のちょうど逆方向のデータフロー。)

### 技術アプローチ

- **取得**: 別アプリからの読み込みなのでREST必須。`GET /k/v1/records.json`(条件クエリ + `fields`指定、共通前提の`$id`昇順ページングで全件取得)。
- **書き込み**: レコード追加/編集画面で実行するなら`kintone.app.record.get()`/`set()`でサブテーブル(`SUBTABLE`)のvalue配列に行を積む(JS APIで完結、保存はユーザー操作)。詳細画面から直接更新するなら`PUT /k/v1/record.json`(既存行をすべて含めないと消える点に注意)。
- 検索条件文字列の組み立ては構想1・4と共通の条件ビルダー/クエリ合成ロジックを流用(コピー)できる。
- **検索条件パラメータ(確定)**: 検索条件の値部分に固定値だけでなく「現在のレコードのフィールド値」を差し込める(例: 自レコードの`顧客コード`と一致する取得元レコードのみ取り込む)。クエリ合成ロジックに、値ソースが`固定値`か`自レコードのフィールド参照`かを切り替えるパラメータを持たせ、`src/js/lib/`の純粋関数としてTDD(構想1・4のクエリ合成と統一的な設計にする)。
- **既存行の扱い(確定)**: 全置換。実行のたびにサブテーブルをクリアし、取得結果で上書きする(`kintone.app.record.set()`でサブテーブルのvalue配列を丸ごと差し替え)。

### 設定画面スケッチ

- 取得元アプリID
- 検索条件(フィールド・演算子・値。値は固定値 or 自レコードのフィールド参照を選択)
- フィールドマッピング(取得元フィールド → サブテーブル列)
- 実行トリガー(ボタン設置画面)

### 複雑性・リスク・未決事項

- **実行タイミングがメモにない**: 追加/編集画面のボタンか、レコード表示時の自動取得か。要確認(ボタン起点が安全)。
- 取り込み件数の上限。公式ドキュメントに「1つのテーブルに大量の行を追加しない」旨の注意があるため、上限(例: 数百行)とヒット件数超過時の挙動を決める必要がある。
- 型の不一致(取得元フィールドとサブテーブル列の型互換)の検証をどこまで設定画面で行うか。

### 規模感: **中**(スコープ確認次第で小〜中)

単方向のデータ取り込みで構造は単純。構想1より先に作ると、クロスアプリ取得・マッピングUIの土台になる。

## 6. 神エクセル帳票印刷プラグイン

### 概要(整理した要件)

- Excelテンプレートのセル番地とkintoneフィールドを対応付け、レコードの値を流し込んだExcelファイルを生成・ダウンロードする。
- サブテーブルは別シートに、行範囲を指定してカラム範囲ごとに繰り返し挿入する。
- 可能であればPDFでのダウンロードにも対応する。
- 複数レコードの一括ダウンロードに対応する。**確定**: 一括ダウンロードはzipライブラリ(fflate等)を追加導入し、まとめて1つのzipファイルとしてダウンロードする(連続個別ダウンロードにはしない)。
- PDF出力は初期スコープから除外し、xlsxのみとする(確定・ユーザー回答済み。計画書の示唆通り)。
- **例外事項**: 本プラグインに限り、「安全が確認されたバージョンの」外部Excel操作ライブラリ・zip操作ライブラリの利用が認められている(CLAUDE.md開発方針9「外部パッケージを使わない」の例外。当初はExcel操作ライブラリのみを想定していたが、一括ダウンロードのzip化のためzipライブラリにも例外範囲を拡張する)。ただし**外部サーバーとの通信禁止は維持**する — ライブラリはビルド時にバンドルし、CDN読み込みや外部変換サービスは使わない。採用する全ライブラリ(Excel操作・zip操作)について「安全が確認された」の要件(下記)を満たすこと。

### 「安全が確認された」の具体化(secureCodingGuideline.md準拠)

secureCodingGuideline.mdの「外部ライブラリ利用時はリスクを考慮する」に沿って、少なくとも以下を満たすこと:

- 信頼できる開発者・組織が管理し、メンテナンスが継続している(更新が止まったパッケージは不可)
- 依存関係が最小であること(推移的依存の多いライブラリは避ける)
- `pnpm audit`で既知脆弱性がないバージョンに**固定**(lockファイル + 完全固定バージョン指定)し、意図しない自動更新を防ぐ
- 重大な脆弱性が報告された場合の更新・代替手順をidea.md/security-checklistに明記する
- 追加の運用として: 採用バージョンとその選定理由・監査日をsecurity-checklistに記録し、バージョンを上げるたびに再監査する

### 技術アプローチ

- **ライブラリ候補**: ExcelJS または SheetJS(xlsx)。**重要な選定基準は「テンプレートの書式(罫線・結合セル・列幅)を保持したまま読み書きできるか」** — SheetJS Community Editionはスタイルの書き出しに制約があり(スタイル対応はPro版)、神エクセル用途では致命的になり得る。ExcelJSは書式保持に比較的強いが依存とサイズが大きい。**実装前に両者で実テンプレートの読み込み→値差し込み→書き出しの検証(スパイク)を行うべき**。
- **データ取得**: 自レコードは`kintone.app.record.get()`、一括ダウンロード時は`kintone.app.getQuery()` + `GET /k/v1/records.json`(REST)。
- **テンプレート保存(確定)**: プラグイン設定へのbase64保存は行わない。**対象アプリに限らない任意のアプリの添付ファイルフィールド**にあるファイルをテンプレートとして指定する。添付ファイルフィールドには複数ファイルを添付できるが、**1番目のファイル**をテンプレートとして採用する。設定は「アプリID + レコードID + フィールドコード」の3点で指定し、実行時に`GET /k/v1/record.json`でレコードを取得→対象フィールドの1番目のファイルの`fileKey`を得て`GET /k/v1/file.json`でダウンロードする。256KB(setConfig)制限を受けないため、大きめのテンプレートにも対応できる。
- **ダウンロード**: 生成したxlsx(単一時)はBlob + `a[download]`でクライアント完結。一括時は各xlsxをメモリ上に生成し、zipライブラリでまとめて1つのzipとしてBlob + `a[download]`でダウンロードする。

### 設定画面スケッチ

- テンプレートの参照先(アプリID + レコードID + 添付ファイルフィールドのフィールドコード。フィールドの1番目のファイルを使用する旨を明記)
- セルマッピング(シート名 + セル番地 ⇔ フィールドコード)の一覧編集
- サブテーブル設定: 対象テーブル / 出力先シート / 繰り返し開始行と行範囲 / 列⇔テーブル列の対応
- 出力ファイル名の規則(フィールド値の差し込み)
- 一括ダウンロードの上限件数

### 複雑性・リスク・未決事項

- 神エクセル特有の複雑なテンプレート(多段結合セル・図形・印刷設定)がライブラリの読み書きで壊れるリスク。サブテーブルの行挿入でテンプレート下部のレイアウトが崩れる問題も典型。
- テンプレート参照先レコード・フィールドが削除/変更された場合のエラーハンドリング(添付ファイル参照方式に変更したことで生じる新たな未決事項)。
- マッピング設定UIの作り込み(セル番地の手入力か、プレビュー付きか)で工数が大きく変わる。
- 一括ダウンロード時のメモリ使用量(ブラウザ内で数十ファイル分のxlsx生成+zip圧縮)。

### 規模感: **大**(かつ6案中で最も不確実性が高い)

外部ライブラリ選定・テンプレート書式保持・PDF可否という「作ってみないと分からない」要素が集中している。

## 実装順序の提案(要確認)

以下はあくまで提案であり、優先度は事業判断で入れ替えてほしい。

1. **西暦和暦変換(小)** — 純粋ロジック+JS APIのみで完結し、`fiscal_year_numbering`と同型。TDD・E2Eの型を最小コストで一周できる。
2. **複数レコードサブテーブル化(中)** — クロスアプリのREST取得・フィールドマッピングUI・条件ビルダーという、構想1と4で必要になる土台を最小構成で先に作れる。
3. **サブテーブル別アプリ挿入(大)** — 2の逆方向。マッピングUI・条件エンジンを流用しつつ、`kintone.createDialog()`・UPSERT・結果表示を積み増す。
4. **関連レコード集計(中〜大)** — 3までで作ったクエリ合成・REST分割実行の経験の上に、確認ダイアログ・ローディング・一括実行のパターンを確立する。
5. **ガントチャート(大)** — 技術的には他とほぼ独立(描画が主戦場)なので順序の自由度が高い。UI工数が大きいためまとまった期間を確保できるタイミングで。
6. **神エクセル帳票印刷(大・高リスク)** — 唯一の外部ライブラリ例外であり、リポジトリの方針(監査・バージョン固定の運用)を先に固める必要がある。最後に回す案を基本としつつ、**「ExcelJS/SheetJSで実テンプレートの書式が保持できるか」「PDFを諦めるか」の2点だけは早期に小さなスパイクで検証しておく**と、後の手戻りを防げる。

順序の考え方: 2→3→4 は「クロスアプリREST + 条件/マッピングUI」という共通の技術的地盤を段階的に厚くする並び。1は地盤に依存しない即戦力、5は独立、6はリスク先出し(スパイク)と実装本体(最後)を分ける、という整理。
