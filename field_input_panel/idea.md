# 入力補助サイドパネルプラグイン(field_input_panel)

元メモ・元プラグイン名は「入力箇所モーダル表示プラグイン」だったが、実装がモーダルからサイド
パネル方式に変わったため、ユーザー確認のうえプラグイン名を「入力補助サイドパネルプラグイン」に
変更した(確定)。

## 機能

レコード追加・編集画面にフローティングボタンを設置し、押下すると画面右端に固定表示される入力パネルが
開く。パネルには設定済みの特定フィールドだけが1列で並び、まとめて入力できる。背景の元フォームは隠れず
見えたままなので、他のフィールドの値を確認しながら入力できる(元メモは「モーダル」だったが、ユーザー
確認の結果、まず「画面下1/4くらいに出して、他のフィールドを見ながら入力できるように」、続けて
「右のコメント欄のところに1列で並べる」という要望に変更、確定。最終形は画面右端に固定表示・1列レイアウト)。
パネルの「反映」ボタンで`kintone.app.record.set()`により実フォームに値を反映し、「閉じる」でパネルを
閉じる(反映せずに閉じても入力中の値は破棄されるだけで、実フォーム側には影響しない)。

## 対象画面(確定・元メモの「詳細画面」から変更)

`kintone.app.record.set()`はPC/モバイルともレコード**追加画面**・**編集画面**でのみ利用可能で、
レコード**詳細画面**(閲覧のみ)では利用できないことをkintoneドキュメントMCPで確認済み。元メモの
「詳細画面にフローティングボタンを設置」は、実際に値を書き換えられる画面という意味で
`app.record.create.show` / `app.record.edit.show`(レコード追加画面・編集画面を表示した後のイベント)
を対象とした。

## パネルの実装方式(確定)

`kintone.createDialog()`(公式のダイアログ作成API、PC専用)は中央にオーバーレイ表示され背景を覆うため、
「他のフィールドを見ながら入力したい」という要望に合わない。そのため本プラグインでは
`kintone.createDialog()`を使わず、`position: fixed`で画面右端に固定表示する独自のパネル(vanilla JS+CSS)
を実装する。パネルは背景をダイム(暗転)させず、フォーム側のクリック・スクロールも妨げない。

kintoneには、レコード詳細・編集画面の右サイドバー(レコードコメント/変更履歴)の開閉を制御するAPI
(`kintone.app.record.showSideBar()`/`getSideBarDisplayState()`)はあるが、そのサイドバー**内部に
任意のコンテンツを挿入する公開APIは存在しない**ことをkintoneドキュメントMCPで確認済み。また
サイドバー自体はレコード詳細・編集画面専用で、レコード追加画面には存在しない。そのため本プラグインは
kintoneの実サイドバーを操作・流用せず、同じ幅・同じ右端の位置に**独自のDOM要素を`z-index`で重ねて
覆いかぶせる**方式にした(ユーザー確認: 「コメント欄に入れる訳じゃなくて、コメント欄の上にDOM被せる。
幅がそうなだけで、基本メニューも隠れていい」)。レコード追加画面(サイドバー自体が無い画面)でも同じ
位置・幅で一貫して表示できる。

- フローティングボタン: 画面左下に`position: fixed`で表示(右端はパネルの表示位置と重なるため左下にした)。
  ボタンを複数設定した場合は縦に積み上げる。
- パネル: `position: fixed; top: 0; right: 0; bottom: 0`、幅はkintoneの標準的なコメントサイドバー幅を
  目安に`360px`程度(狭い画面では`90vw`まで縮小)。ヘッダー上部からビューポート下端まで覆う(コメント
  サイドバーだけでなくkintoneのヘッダーメニューの右側が隠れても構わない、との確認済み)。`z-index`を
  高く設定し、他の要素の上に重ねて表示する。ヘッダー(タイトル・閉じるボタン)、本文(1列の入力項目、
  縦スクロール可)、フッター(反映・閉じるボタン)の3段構成。
- 同時に開けるパネルは1つ(複数のフローティングボタンがあっても、ボタン押下で既存パネルを閉じて
  差し替える)。

## 対応フィールド型(確定)

`kintone.app.getFormFields()`で取得したフィールドのうち、以下の型のみ選択・表示できる。

| 対応 | フィールド型 |
| :-- | :-- |
| ○ | 文字列(1行)・文字列(複数行)・数値・リンク・ラジオボタン・チェックボックス・複数選択・
ドロップダウン・日付・時刻・日時 |
| ✕ | ルックアップ(`lookup`プロパティを持つフィールド。コピー元のフィールド型に関わらず除外)、
ユーザー選択・組織選択・グループ選択、添付ファイル(`kintone.app.record.set()`の制限事項により
書き換え不可)、リッチエディター(HTML編集はスコープ外)、関連レコード一覧・テーブル(値の取得/登録が
できない、またはスコープ外)、計算・レコード番号・作成者・作成日時・更新者・更新日時・ステータス・
作業者・カテゴリー(システム項目、値の登録/更新不可) |

元メモの「ユーザー・組織・グループ(ロール)・ルックアップフィールドは標準では実装が難しそう」の通り、
これらは選択肢に出さない。`js/lib/field-eligibility.js`の`isEligibleField()`で判定する。

## 初期値の取得と反映(確定)

- パネルを開いたときに`kintone.app.record.get().record`から各フィールドの現在値を読み込み、
  入力コントロールへ反映する(`js/lib/field-value-codec.js`の`decodeFieldValue()`)。
- 「反映」ボタン押下時に`kintone.app.record.get().record`を取得し直し、パネルの入力値を
  `encodeFieldValue()`で各フィールド型の値形式に変換して`record[fieldCode].value`に代入したうえで
  `kintone.app.record.set({ record })`を呼ぶ(取得したレコードオブジェクトを直接ミューテートして
  そのまま渡す。`type`キー等は元のオブジェクトに既に入っているため自前で組み立てない。org_lookupの
  `applyValuesToRecord()`と同じパターン)。
- フローティングボタンのクリックイベントは`kintone.events.on()`のハンドラーの外側(通常の
  `addEventListener('click', ...)`)なので、`record.get()`/`record.set()`の呼び出し制限
  (イベントハンドラー内では実行不可)を受けない(org_lookup/self_lookupと同じ理由)。

### 日時(DATETIME)フィールドのタイムゾーン変換(確定)

kintoneのDATETIME値はUTCのISO8601文字列(例: `2012-01-11T11:30:00Z`)。入力コントロールには
`<input type="datetime-local">`を使うため、パネル表示時は`Date`オブジェクトでブラウザのローカル
タイムゾーンに変換して`YYYY-MM-DDTHH:MM`形式にし、反映時は逆変換してUTCのISO8601文字列に戻す
(`js/lib/field-value-codec.js`)。日付・時刻フィールドはタイムゾーンを持たない値([日付のフォーマット]
のページで日時のみがタイムゾーン変換の対象と確認済み)なのでそのまま文字列を扱う。

## 1列レイアウトと空白挿入(確定・元メモの「2列」から変更)

各ボタンの項目リストは「フィールド」または「空白」の並びを持ち、設定した順に縦1列でそのまま流し込む
(元メモは「2列」だったが、パネルが右端の縦長エリアになったため、ユーザー確認の結果1列に変更)。
「空白」は1項目分の空行として描画され、関連するフィールドをグループごとに視覚的に区切るための
調整に使う(元メモ「切りよく改行するための空白も挿入可」を、1列レイアウトでの「空行」に読み替えた)。

項目の並び替えは各行の「↑」ボタンで前の項目と入れ替える方式とした(ドラッグ&ドロップは他プラグイン
(`hierarchy_view`)のスコープであり、本プラグインでは過剰と判断)。

## 設定画面(確定)

`kintone.plugin.app.setConfig()`にのみ保存する。ボタンを追加するたびに設定画面上部にタブが増える
UI(元メモ「ボタン追加ごとにタブが増えていくような形」)。

- タブ見出し: ボタンのラベル(未入力時は「ボタンN」)+削除ボタン。クリックで表示するボタンを切り替える。
- 選択中のボタンの設定:
  - フローティングボタンに表示するラベル(必須、短い文言を想定)
  - パネルのタイトル(省略可、未入力時はボタンラベルを使う)
  - 項目リスト(複数追加・削除できる)。行ごとに「フィールドを選ぶ」(対応フィールド型のみ選択肢に
    出す)か「空白を挿入」を選べる。フィールド行は同じボタン内で重複選択不可。
- 保存時に`js/lib/config-validation.js`でチェックする(ボタンのラベル未入力、項目0件、フィールド
  未選択のFIELD項目、同一ボタン内でのフィールドコード重複)。

## TDD

`src/js/lib/`配下の純粋ロジックはJestでユニットテストする(`pnpm test`)。

- `field-eligibility.js` — フィールドのプロパティ(型・`lookup`設定の有無)から、本プラグインで
  選択可能なフィールドかどうかを判定する`isEligibleField()`
- `field-value-codec.js` — フィールド型ごとに、`record.get()`の値→入力コントロール用の値
  (`decodeFieldValue()`)、入力コントロールの値→`record.set()`用の値(`encodeFieldValue()`)を
  相互変換する。DATETIMEのタイムゾーン往復変換、チェックボックス/複数選択の配列、日付・時刻の
  空値(`null`)への変換を含む
- `config-store.js` — `kintone.plugin.app.getConfig()`/`setConfig()`のペイロードの読み書きと
  デフォルト値
- `config-validation.js` — 設定(ボタンの配列)のバリデーション

kintone依存のグルーコード(`desktop.js`/`config.js`、DOM描画・`kintone.app.getFormFields()`/
`record.get()`/`record.set()`呼び出しを含む)は`src/e2e/*.e2e.test.js`(Puppeteer、
`pnpm run test:e2e`)で実環境テストする。

## モバイル非対応(確定)

画面右端に固定幅パネルを重ねる方式は、画面幅の狭いモバイル端末では大部分(またはすべて)を覆ってしまい
「他のフィールドを見ながら入力する」という目的に合わなくなる。ソフトウェアキーボード表示時の重なりなど
も別途検証が必要になるため、`confirm_modal`/`status_arrow`と同様、今回のスコープではPC専用とした
(`mobile.js`は作成しない)。

## 実装

kintoneドキュメントMCPを参照しながら実装した。`kintone.app.record.set()`がレコード追加・編集画面
限定であること、`kintone.app.getFormFields()`の戻り値がREST APIの`properties`と同様の値である(=
`{ properties: {...} }`のようにラップされない)こと、フィールド型ごとの値の形式・空値の扱い
([フィールド形式](/id/2736678ef8d2aad09a33e8bb/))を確認済み。セキュアコーディングガイドラインでの
リスクチェックは`security-checklist.md`を参照。
